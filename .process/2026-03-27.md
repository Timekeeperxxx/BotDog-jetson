## 二、按实现顺序的开发过程

### 步骤 1：修复架构底层 DDS Topic 生成失败的致命 Bug
- 修改目标：开发板上启动服务时持续抛出 `[DDS_RETCODE_PRECONDITION_NOT_MET]` 错误，导致系统瘫痪无法创建 DDS 通信节点。
- 涉及文件：`backend/workers_unitree_telemetry.py`, `scripts/start_backend.sh`
- 主要改动：
  - 移除了 Python 代码中重复定义的 `ChannelSubscriber` 初始化导致的句柄冲突。
  - **核心修复**：定位到是 pip 预编译的 CycloneDDS C 库在 ARM64 上存在 Bug。修改 `start_backend.sh` 脚本，在启动 Python 进程前强行注入 `CYCLONEDDS_HOME` 与 `LD_LIBRARY_PATH`，迫使应用加载用户手工编译的健康版 C 库层，并适配了从 ens37 到 eth0 的默认网卡切换。
- 实现结果：DDS 通信恢复正常，机器狗底层运动数据收发通道正式打通。
- 备注：这是一个阻塞全系统的架构级难题。

### 步骤 2：自动跟踪能力默认开启与前端瘦身
- 修改目标：排查自动跟踪“只能检测不出判定”的问题，并精简前端无用的开发期调试面板。
- 涉及文件：`backend/config.py`, `frontend/src/components/AutoTrackPanel.tsx`
- 主要改动：
  - 排查发现跟踪功能总闸 `AUTO_TRACK_ENABLED` 默认为 False，导致系统仅在运行单纯越界报警流。将其改为 True 使功能开箱即用。
  - 在前端面板中去除了大块的“实时决策日志”与冗杂的目标运动状态明细，仅保留控制启停和人员标记功能，让画面更专注。
- 实现结果：跟踪判定开始运行，且前端界面回归到清爽的监控形态。

### 步骤 3：跟踪控制参数全量接管到环境变量
- 修改目标：方便现场在开发板上通过改文件热调优，而不是走繁琐的数据库配网流程。
- 涉及文件：`backend/main.py`, `backend/config.py`
- 主要改动：在 `main.py` 的初始化工厂中，去掉了防撞线、目标死区、前进面积比等关键参数强制从 `db_configs` 覆盖的逻辑，统一改为直读 `settings` (`.env`)。
- 实现结果：实现现场即刻改配即刻生效的调参闭环。

### 步骤 4：重构跟踪防撞逻辑，化解“畏缩不前”
- 修改目标：真机测试发现狗距离人很远就停住了，因为目标双脚在广角视野下很容易触碰到底部 80% 的警戒线。
- 涉及文件：`backend/follow_decision_engine.py`, `backend/config.py`
- 主要改动：
  - 将 `AUTO_TRACK_ANCHOR_Y_STOP_RATIO` 从 80% 下压至 90%，大幅度缩小画面底部的停车禁区。
  - 重写决策树判决优先级：**面积绝对优先**。只要目标检测框的面积（`area_ratio`）小于规定大小，引擎就会无视防撞线强制发送 `forward` 向前走，仅在面积达标后才引入双脚位置微调。
- 实现结果：机器狗现在敢于主动缩短距离，紧贴目标直到画面面积达标才刹车。

### 步骤 5：调优命令频率与死区，解决转向“机械感”
- 修改目标：用户反馈转弯和直行都显得迟钝、机械。
- 涉及文件：`backend/config.py`
- 主要改动：
  - 将转弯死区 `AUTO_TRACK_YAW_DEADBAND_PX` 从 80 像素拦腰砍成 40 像素，视野偏离一点就会转头纠正。
  - 将命令发送节流限制 `AUTO_TRACK_COMMAND_INTERVAL_MS` 从 200ms 大幅缩短至 100ms，让决策下发速度翻倍。
- 实现结果：显著提高了跟拍时的响应跟脚程度，动作连接更加丝滑流畅。
