# BotDog 视频流系统 - 完整修复说明

## 问题诊断

### 错误现象
```
OSError: [WinError 10049] 在其上下文中，该请求的地址无效
```

### 根本原因
1. **配置错误**: `UDP_RELAY_TARGET_ADDRESS` 被错误设置为 `"127.0.1"`（缺少最后一位）
2. **架构冲突**: UDP 转发器试图监听端口 5000，但 `video_track_native.py` 也需要监听同一端口
3. **设计重复**: 新架构中，GStreamer 直接监听 UDP 端口，不需要转发器

### 解决方案

#### 1. 禁用 UDP 转发器
在 `backend/config.py` 中添加：
```python
UDP_RELAY_ENABLED: bool = False  # 禁用 UDP 转发器
```

#### 2. 修正目标地址
```python
UDP_RELAY_TARGET_ADDRESS: str = "127.0.0.1"  # 修正 IP 地址
```

#### 3. 修改启动逻辑
在 `backend/webrtc_signaling.py` 中：
```python
async def start_udp_relay(self) -> None:
    """启动 UDP 视频流转发器。"""
    if not settings.UDP_RELAY_ENABLED:
        logger.info("UDP 转发器已禁用（video_track_native 直接监听端口）")
        self._udp_relay_started = True
        return
    # ... 原有启动代码
```

## 系统架构（修正后）

```
相机 RTSP → camera_streamer.py → UDP 5000 → GStreamer → WebRTC
192.168.144.25                     (192.168.144.30)    (D3D11)
```

### 数据流说明

1. **相机推流** ([camera_streamer.py](camera_streamer.py))
   ```bash
   rtspsrc location=rtsp://192.168.144.25:8554/main.264
   → rtph265depay → h265parse → rtph265pay
   → udpsink host=192.168.144.30 port=5000
   ```

2. **后端解码** ([video_track_native.py](backend/video_track_native.py))
   ```bash
   udpsrc port=5000
   → rtpjitterbuffer → rtph265depay → h265parse
   → avdec_h265 (自动选择 d3d11h265dec)
   → videoconvert → fdsink fd=1
   ```

3. **WebRTC 推送** ([webrtc_signaling.py](backend/webrtc_signaling.py))
   - 读取 GStreamer stdout 的原始像素数据
   - 转换为 VideoFrame
   - 通过 WebRTC 发送到浏览器
