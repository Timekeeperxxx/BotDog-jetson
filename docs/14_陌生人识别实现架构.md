# BotDog 陌生人识别与跟踪功能实现架构

## 1. 文档目标

本文档给出“陌生人识别与跟踪”功能的可实现架构，要求直接贴合当前 BotDog 仓库结构。  
本文档重点不是算法原理介绍，而是：

1. 哪些现有模块保留
2. 哪些模块新增
3. 哪些模块需要重构
4. 状态机如何设计
5. 接口如何定义
6. 控制仲裁如何落地
7. 前后端如何对齐
8. 如何保证鲁棒性和安全边界

---

## 2. 现有代码架构映射

### 2.1 现有核心模块

#### 2.1.1 `backend/main.py`
作用：
- 生命周期装配
- 初始化状态机
- 初始化遥测队列
- 初始化 MAVLink 网关
- 初始化 AI Worker
- 初始化事件广播器
- 初始化控制服务
- 注册 REST / WebSocket 路由

结论：
- 继续保留
- 新功能仍通过 `lifespan` 装配
- 不建议把新跟踪逻辑直接塞进 `main.py`

#### 2.1.2 `backend/workers_ai.py`
作用：
- FFmpeg 读流
- YOLO 检测
- 稳定命中
- 抓拍
- 调用 AlertService

结论：
- 这是新功能的主扩展点
- 但不能继续把所有逻辑都堆在一个 worker 中
- 需要将其拆出“检测、候选目标、跟踪状态、控制决策”

#### 2.1.3 `backend/control_service.py`
作用：
- 接收控制命令
- 校验命令
- 急停检查
- 限流
- Watchdog 自动 stop

结论：
- 继续保留
- 自动跟踪不应绕过它直接发底层控制
- 新功能要基于此服务实现“自动控制命令通道”

#### 2.1.4 `backend/robot_adapter.py`
作用：
- 统一适配底层控制接口
- 当前仅有模拟适配器和预留 MAVLink 适配器骨架

结论：
- 本期保持接口不变
- 增加 B2 适配器或 velocity 能力时从这里演进
- 自动跟踪层不应依赖具体底盘实现

#### 2.1.5 `backend/state_machine.py`
作用：
- 管理系统级状态
- 管理心跳与急停

结论：
- 继续保留为“系统级状态机”
- 新功能应新增一套“自动跟踪状态机”
- 不建议把所有跟踪状态硬塞进现有 `SystemState`

#### 2.1.6 `backend/alert_service.py`
作用：
- 证据生成
- 事件落库
- 告警广播

结论：
- 继续复用
- 自动跟踪事件统一通过这里发“标准事件”
- 不要在各模块里直接重复广播和落库

#### 2.1.7 `backend/ws_event_broadcaster.py`
作用：
- `/ws/event` 广播

结论：
- 继续复用
- 自动跟踪状态事件统一从这里下发

#### 2.1.8 `backend/models.py`
作用：
- 任务、证据、日志、配置实体

结论：
- 可继续承载事件记录
- 本期建议新增少量表，而不是大规模重构

## 3. 总体架构设计

## 3.1 架构原则

1. 继续沿用当前 FastAPI + Worker + Service 的分层
2. 自动跟踪逻辑不直接侵入 `main.py`
3. AI 检测、目标管理、区域判断、控制仲裁、事件留痕分层
4. 所有自动控制命令统一经过 `ControlService`
5. 人工控制优先级必须高于自动跟踪
6. 跟踪对象始终唯一，候选目标可多
7. 自动跟踪与事件系统必须统一

---

## 3.2 推荐逻辑架构

```text
RTSP Video
  -> AIWorker Frame Loop
      -> Detector
      -> Candidate Extractor
      -> Stranger Rule Evaluator
      -> Target Manager
      -> Zone Evaluator
      -> Follow Decision Engine
      -> Control Arbiter
      -> ControlService
      -> RobotAdapter

同时：
  -> AlertService
  -> EventBroadcaster
  -> Evidence/Logs/DB
  -> Frontend /ws/event
```

---

### 4.2.5 `backend/follow_decision_engine.py`
职责：
- 根据当前目标位置和状态生成自动控制决策

- zone state
- image center
- current tracking state
- lost state
- manual override state

输出：
- `TrackDecision`

  - `command`
  - `reason`
  - `priority`
  - `stop_required`

说明：
- 本期直接产出离散命令：
  - `left`
  - `right`
  - `forward`
  - `stop`
- 后续可扩展速度控制版本

---

### 4.2.6 `backend/control_arbiter.py`
职责：
- 管理控制权归属
- 实现“急停 > 手柄 > 网页 > 自动跟踪”的优先级

输入：
- 手柄活动状态
- 网页人工控制状态
- 自动跟踪决策
- 急停状态
- 自动跟踪开关

输出：
- 最终允许下发的控制命令
- 当前控制拥有者 `control_owner`

说明：
- 自动跟踪不能直接调用 `ControlService`
- 必须先经过仲裁器

---

### 4.2.7 `backend/auto_track_service.py`
职责：
- 统一封装自动跟踪主流程
- 对外提供：
  - enable / disable
  - pause / resume
  - mark target as known
  - stop with reason
  - process frame detections

说明：
- 推荐作为 `AIWorker` 依赖注入的上层服务
- 让 `AIWorker` 负责采帧与检测，`AutoTrackService` 负责业务决策

---

### 4.2.8 `backend/services_tracking_config.py`（可选）
职责：
- 统一读写自动跟踪配置
- 从现有 `SystemConfig` 或 `.env` 补齐新配置项读取逻辑

## 5. AI Worker 改造方案

## 5.1 当前问题

当前 `workers_ai.py` 是“检测即告警”的单路径实现。  
问题是：

1. 只能处理单个最佳目标
2. 不保存目标状态
3. 无多目标选择
4. 无自动跟踪状态机
5. 无人工打断仲裁
6. 无出区停止

## 5.2 改造目标

AIWorker 未来只负责三件事：

1. 读视频帧
2. 调 detector 获取当前帧检测结果
3. 把检测结果交给 `AutoTrackService`

## 5.3 新流程建议

```text
AIWorker.start()
  -> read frame
  -> detector.detect_many(frame)
  -> auto_track_service.process_frame(frame, detections, telemetry_context)
  -> if need evidence:
         alert_service.handle_ai_event(...)
```

## 5.4 不建议的做法

不建议继续在 `workers_ai.py` 中不断堆：

- 多目标排序
- 区域判断
- 控制优先级仲裁
- 会话已知人员逻辑
- 前端状态事件构造

这些应该拆出单独模块。

## 6. 跟踪状态机设计

## 6.1 系统状态机与自动跟踪状态机分离

### 系统状态机
继续使用现有 `StateMachine`：
- `DISCONNECTED`
- `STANDBY`
- `IN_MISSION`
- `E_STOP_TRIGGERED`

### 自动跟踪状态机
新增 `AutoTrackState`：

- `DISABLED`
- `IDLE`
- `DETECTING`
- `TARGET_LOCKED`
- `FOLLOWING`
- `OUT_OF_ZONE_PENDING`
- `LOST_SHORT`
- `MANUAL_OVERRIDE`
- `PAUSED`
- `STOPPED`

---

## 6.2 状态迁移

### 6.2.1 `DISABLED`
自动跟踪总开关关闭。

迁入条件：
- 自动跟踪未启用
- 任务结束
- 系统未进入可控状态

迁出条件：
- 开关打开且系统进入 `IN_MISSION`

### 6.2.2 `IDLE`
无当前目标，等待目标进入。

### 6.2.3 `DETECTING`
检测到人员候选，但还未满足陌生目标锁定条件。

### 6.2.4 `TARGET_LOCKED`
已选定 `target_id`，但尚未开始发跟随控制。

### 6.2.5 `FOLLOWING`
允许发自动控制命令。

### 6.2.6 `OUT_OF_ZONE_PENDING`
目标刚出重点区，等待确认。

### 6.2.7 `LOST_SHORT`
短时丢失目标，进行轻量等待或微搜索。

### 6.2.8 `MANUAL_OVERRIDE`
人工控制接管。

### 6.2.9 `PAUSED`
自动跟踪手动暂停。

### 6.2.10 `STOPPED`
本轮目标跟踪结束，等待回到 `IDLE`

## 7.2 目标选择策略

### 保持优先
如果当前 active target 仍满足：

1. 仍在画面中
2. 仍在重点区内
3. 未失效
4. 未被人工标记为已知

则默认保持跟踪，不切换。

### 切换条件
只有在以下任一情况出现时才允许切换：

1. 当前目标丢失
2. 当前目标被标记为已知
3. 当前目标出区结束
4. 新目标分数显著高于当前目标
5. 当前目标跟踪状态异常

### 分数模型
建议采用加权分数：

- 核心区加权
- 距离近加权
- 框面积大加权
- 置信度加权
- 时间连续性加权

## 8. 控制架构设计

## 8.1 当前控制基线

当前 `ControlService` 已支持：
- 合法命令校验
- 急停拒绝
- 限流
- Watchdog 自动 stop

因此自动跟踪不得绕开 `ControlService`。

---

## 8.2 控制仲裁设计

### 控制拥有者 `ControlOwner`
建议枚举：

- `NONE`
- `AUTO_TRACK`
- `WEB_MANUAL`
- `REMOTE_CONTROLLER`
- `E_STOP`

### 优先级
- `E_STOP`
- `REMOTE_CONTROLLER`
- `WEB_MANUAL`
- `AUTO_TRACK`

### 仲裁规则
1. 急停存在时，所有自动命令作废
2. 手柄有活动输入时，自动跟踪立即暂停
3. 网页人工控制有活动输入时，自动跟踪立即暂停
4. 自动跟踪只有在无人接管时才能发命令

---

## 8.3 自动跟踪控制输出

### 本期输出
离散动作命令：
- `left`
- `right`
- `forward`
- `stop`

### 决策依据
- `err_x = target_center_x - image_center_x`
- 若目标偏左：`left`
- 若目标偏右：`right`
- 若目标居中且较远：`forward`
- 若目标距离合适：`stop`
- 若出区 / 丢失 / 被打断：`stop`

## 12. 鲁棒性设计

## 12.1 断流处理
RTSP 断流时：
- `AIWorker` 进入降级
- `AutoTrackService` 立即 stop
- 广播 `AUTO_TRACK_INTERRUPTED(reason=video_stream_lost)`

## 12.2 数据库存储失败
- 证据落库失败不得阻塞控制主流程
- 广播失败不得阻塞控制主流程
- 控制仲裁和 stop 必须优先执行

## 12.3 目标抖动
- 增加 `stable_hits`
- 增加 `target_hold_seconds`
- 增加出区防抖 `N_OUT`

## 12.4 命令风暴
- 自动跟踪命令发频需受控
- 通过 `auto_track_command_interval_ms` 节流
- 继续复用现有 `ControlService` rate limit

## 12.5 安全停机
无论何种异常，必须能落到：
- `ControlService.handle_command("stop")`
或
- adapter.stop()

## 13. 基线修复建议

在开发自动跟踪之前，建议先完成以下修复：

1. **修复 `ControlService` 的 E-STOP 状态读取**
2. **统一 `confidence` 取值语义**
3. **统一 `EvidenceItem` 空值约束**

## 15. 结论

对当前 BotDog 来说，最合理的实现方式不是推翻已有系统，而是：

1. 保留现有 AI 检测、控制服务、事件广播、证据留痕底座
2. 在 `AIWorker` 与 `ControlService` 之间增加“目标管理 + 跟踪状态机 + 控制仲裁”
3. 将“陌生人识别”限定为本期可落地的安保规则判断
4. 将“已知人员”限定为本期可落地的会话级人工标记
5. 把自动跟踪设计为可开关、可打断、可留痕、可扩展的能力层

这样既能在当前项目上快速落地，又不会堵死未来接入人脸识别、ReID、B2 速度控制的演进路径。
