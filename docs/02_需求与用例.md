# 智能巡检系统需求与用例概述 (SRS & Use Case Specification)

## 1. 项目概述

### 1.1 背景说明

* ​**行业/场景背景**​：在电力电网、化工园区等高危或极端工业环境中，传统人工巡检面临生命安全风险高、环境适应性差、巡检数据主观性强且难以数字化追溯等问题。
* ​**当前痛点**​：现有商用机器狗大多仅提供基础的局域网遥控器，缺乏与后端中枢系统联动的“天地一体化”数据链路，难以实现超低延迟的远程沉浸式操控，且缺乏边缘端的实时 AI 隐患识别与固化取证能力。

### 1.2 项目目标

* ​**本期必须达成的业务目标（可量化）**​：
  1. 实现 4K 高清视频的 WHEP 端到端推流，全链路延迟严格控制在 ​**< 200ms**​。
  2. 实现 MAVLink 2.0 控制指令的下发与回执，端到端操控延迟 ​**< 150ms**​。
  3. 异常识别（如红外 T\_MAX > 60°C）触发截帧并落库的时效性 ​**< 1000ms**​。
  4. 遥测数据（姿态、经纬度、电量）实现 100% 结构化降采样留存记录。

### 1.3 范围说明

* ​**本期包含的功能**​：单机远程视频监控、HUD 姿态全息显示、基础遥控动作下发、红外热特征异常 AI 报警、SQLite 本地历史轨迹与操作日志留存。
* ​**明确不包含的功能（放到后续版本）**​：多机台蜂群协同调度（Swarm Control）、基于 3D LiDAR 的室内自主 SLAM 构图与无 GPS 导航、云端 PostgreSQL 数据同步。

### 1.4 术语与名词约定

* ​**Session（会话）**​：一次前端终端与后端服务、机器狗之间从“连接建立”到“连接关闭”的连续在线控制会话，对应接口 `/api/v1/session/start` / `/api/v1/session/stop`。
* ​**Task（巡检任务）**​：一次有业务语义的巡检过程（如“西厂区晚班巡检”），在实现上与 Session 一一对应，并在数据库表 `inspection_tasks` 中以 `task_id` 标识。
* ​**Alert（告警事件）**​：后端判定满足风险阈值条件（如 `T_MAX > thermal_threshold`）后产生的事件，统一通过 WebSocket 消息类型 `ALERT_RAISED` 推送。
* ​**Telemetry（遥测数据）**​：由 MAVLink 报文解析出的姿态、位置、电池等状态数据，统一通过 WebSocket 消息类型 `TELEMETRY_UPDATE` 广播。

## 2. 角色与场景

### 2.1 系统角色

* ​**操作员 (Operator)**​：系统核心使用者。负责日常登录终端，实时监控视频画面，下发云台或机体控制指令，并对突发告警进行第一响应（如紧急制动）。
* ​**管理员 / 数据审核员 (Admin / Reviewer)**​：负责资产配置（如阈值修改）、巡检结束后的异常证据链（图片、GPS 坐标、时间戳）复核确认，以及系统级错误日志的回溯排障。
* ​**第三方系统（如有）**​：本期暂无，预留标准化 RESTful API 供未来 ERP/MES 系统对接。

### 2.2 使用场景

* **场景 A：日常巡检**
  * ​**简要背景**​：操作员按既定路线远程驾驶机器狗巡视厂区。
  * ​**期望结果**​：操作员获得无卡顿的沉浸式操控体验，HUD 准确反映狗体物理姿态，保障行驶安全。
* **场景 B：告警复核**
  * ​**简要背景**​：机器狗在巡检中触发了高温阈值，系统自动落盘证据。
  * ​**期望结果**​：审核员在事后能根据数据库检索，精准定位告警发生的物理坐标并查看高清抓拍原图。
* **场景 C：应急排障**
  * ​**简要背景**​：无线链路突发严重干扰，机器狗与地面站断联。
  * ​**期望结果**​：系统秒级感知失联，UI 锁定保护，同时机器狗底层固件无缝接管，执行原地驻留（BRAKE）或返航（RTL）。

## 3. 功能需求

### 3.1 控制终端（前端）功能

#### 3.1.1 登录与权限（如需要）

* ​**触发条件**​：用户通过浏览器访问系统域名。
* ​**主流程**​：输入工号凭证 -> 后端 JWT 鉴权 -> 根据角色加载对应 Dashboard。
* ​**异常处理**​：Token 过期自动跳转登录页，并记录鉴权失败日志。

#### 3.1.2 设备连接与状态总览

* ​**触发条件**​：成功进入主控制台。
* ​**主流程**​：前端发起 WebSocket 连接 -> 订阅遥测 Topic -> 实时更新电量、信号强度、链路延迟等状态位。
* ​**异常处理**​：WebSocket 断开时，触发自动指数退避重连，并在 UI 顶部展示“连接断开”警示条。

#### 3.1.3 机器狗遥控与模式切换

* ​**触发条件**​：操作员通过 FT24 硬件链路输入控制信号。
* ​**主流程**​：控制由硬件链路直达 MCU/飞控，后端仅处理遥测与告警。
* ​**异常处理**​：若当前处于“失联保护”或“极低电量”状态，前端硬拦截指令下发并弹窗提示。

#### 3.1.4 视频监控与全屏/多视角切换

* ​**触发条件**​：点击视频容器的“全屏”按钮。
* ​**主流程**​：通过 WHEP 建立播放 -> `<video>` 标签渲染 -> 响应全屏事件，隐藏侧边栏，保留核心 HUD 叠层。
* ​**异常处理**​：检测到视频卡顿时提示用户检查 FFmpeg/MediaMTX 状态并重连。

#### 3.1.5 告警提示与确认

* ​**触发条件**​：收到后端下发的 `ALERT_RAISED` 报文。
* ​**主流程**​：前端播放警报音 -> 侧边栏“实时抓拍”队列压入新卡片 -> UI 边框红色频闪 -> 操作员点击“确认”。
* ​**异常处理**​：操作员长时间未确认，告警状态应持续保持，并记录入日志表。

#### 3.1.6 历史日志与抓拍浏览

* ​**触发条件**​：点击“历史存档”模块。
* ​**主流程**​：向后端发起 GET 请求获取 `anomaly_evidence` 列表 -> 渲染分页表格及缩略图。
* ​**异常处理**​：请求超时则提示“获取归档数据失败”。

### 3.2 后端服务功能

* ​**MAVLink 网关**​：异步监听 UDP/串口比特流，执行包头校验（0xFD）、解包、过滤高频冗余数据，并将控制指令打包签名下发。
* ​**WebSocket 广播**​：维持前端 Client 池，按照 `TELEMETRY_UPDATE` 规范对心跳、姿态、事件进行 JSON 序列化并以 \~15Hz 广播。
* ​**日志与抓拍落库**​：监听业务流，将 AI 引擎的截帧路径与经纬度元数据通过 SQLAlchemy (ORM) 异步写入 SQLite。
* ​**系统监控与自检**​：启动时校验 MediaMTX/FFmpeg 可用性，运行时监控 MAVLink 进程健康度。

### 3.3 流媒体服务功能

* ​**码流接入**​：FFmpeg 拉取相机 RTSP(H.264/H.265) 并推送到 MediaMTX，前端通过 WHEP 播放。
* ​**弱网重连**​：FFmpeg 断流自动重连，前端提示并允许手动刷新播放。

### 3.4 边缘端接口需求

* ​**飞控固件**​：ArduPilot/PX4 必须开启 `TELEM1/2` 端口的 MAVLink 2.0 协议输出，波特率 ≥ 115200。
* ​**图传/网络**​：提供具备公网 IP 的 5G CPE 或局域网高带宽微波电台，确保上下行带宽对称且 UDP 端口畅通。

## 4. 非功能性需求

### 4.1 性能指标

* ​**端到端视频延迟目标**​：≤ 200 毫秒（从相机曝光到屏幕刷新）。
* ​**控制指令往返延迟**​：控制链路由 FT24 硬件直连，Web 侧不再承载控制延迟指标。
* ​**最大并发连接数**​：单台后端服务器支持至少 50 个 WebSocket 订阅客户端并发。

### 4.2 可用性与可靠性

* ​**年可用性目标**​：99.9% (工业级基础标准)。
* ​**宕机/重启恢复时间要求**​：后端服务崩溃后，由 Systemd 守护进程拉起的时间应 ≤ 5 秒，WHEP 播放重连 ≤ 3 秒。

### 4.3 安全性

* ​**认证/鉴权要求**​：API 访问需携带 Bearer JWT Token。
* ​**通讯加密要求**​：敏感指令及 WHEP 媒体流通过 DTLS/SRTP 进行端到端加密，禁止明文传图。

### 4.4 可维护性与可扩展性

* ​**模块化拆分**​：前端 View 与 Zustand Store 强解耦；后端 FastAPI 路由按业务域划分（HTTP 统一前缀 `/api/v1/...`，WebSocket 统一前缀 `/ws/...`）。
* ​**日志要求**​：依托 Loguru 实现结构化分级（INFO/WARN/ERROR），按天自动 Rotate 轮转。

## 5. 用例规格

### 用例 UC-01：远程巡检控制

* ​**目标**​：操作员安全、流畅地遥控机器狗沿厂区通道行进。
* ​**涉及角色**​：前端操作员。
* ​**前置条件**​：系统正常在线，电池电量 > 20%，前端完成 WebSocket 握手。
* ​**触发事件**​：操作员在 UI 点击“启动任务”，并推移物理操作杆。
* ​**基本流程**​：
  1. 前端按照 10Hz 频率采样手柄位移矢量。
  2. 前端将矢量封装为 JSON 发送至后端 WebSocket。
  3. 后端校验权限后，转化为 `MANUAL_CONTROL` MAVLink 包。
  4. 边缘飞控解析指令，驱动伺服电机执行前进/转向动作。
  5. 前端 HUD 实时展现飞控回传的姿态变化（Pitch/Roll）。
* ​**备选流程 / 异常流程**​：若后端判定 MAVLink 队列拥堵阻塞，丢弃过期控制包并向前端返回限流警告。
* ​**输入数据**​：JoyStick 遥感 X/Y 轴偏置值 (-1000 到 1000)。
* ​**输出数据 / UI 反馈**​：机器狗物理位移；UI 姿态仪及航向角数字实时跳动。
* ​**成功条件**​：狗体动作与画面反馈时延符合预期，无卡顿。
* ​**失败条件**​：指令下发超时或画面冰冻超 1 秒。

### 用例 UC-02：异常温度告警与抓拍

* ​**目标**​：在无人干预下，系统自动发现过热隐患并生成带时空水印的数字证据。
* ​**涉及角色**​：前端操作员、数据审核员。
* ​**前置条件**​：机器狗正处于巡检任务中，热成像模块及 ONNX AI 推理服务正常运行。
* ​**触发事件**​：AI 引擎持续采样流媒体，计算得出某帧最高温度 `T_MAX > 60°C`。
* ​**基本流程**​：
  1. AI 推理管线截取当前原始图像帧，保存至本地 `/data/snapshots/`。
  2. 后端读取最新缓存的 `GLOBAL_POSITION_INT` 获取经纬度。
  3. 后端将 `task_id`、温度极值、经纬度、图像路径组装，插入 `anomaly_evidence` 表。
  4. 后端通过 WebSocket 广播 `ALERT_RAISED` 消息。
  5. 前端操作员界面触发红色频闪与提示音，抓拍列表更新。
* ​**备选流程 / 异常流程**​：若数据库磁盘写满，后端放弃落盘图像，退级为仅记录文字 Log，并向管理员发送 CRITICAL 报错。
* ​**输入数据**​：红外热力图数据流。
* ​**输出数据 / UI 反馈**​：持久化的图文告警记录；UI 强干扰视觉警报。
* ​**成功条件**​：完成完整证据链闭环，人工能在历史界面复核原图。
* ​**失败条件**​：图像损坏或 GPS 元数据丢失。

## 6. 约束与假设

### 6.1 环境约束

* 必须在工业现场部署具备 Qos 保障的 5G 专网或 WiFi-6 覆盖，确保上行带宽稳定在 15Mbps 以上。

### 6.2 硬件约束

* 边缘算力必须基于 NVIDIA Jetson 架构（Nano 或 NX），以调用原生 `nvv4l2h264enc` 硬编单元。
* 机器狗底层必须支持开源 MAVLink 协议簇交互。

### 6.3 项目假设

* 假设在实施区域内具备 RTK 或良好的常规 GPS 信号，以保证抓拍证据链的地理坐标漂移在允许误差内（室内场景本期暂不适用）。

## 7. 版本规划与里程碑

* **版本 v1.0 (MVP) - 第1-2月**
  * 核心功能：单机视频流拉取、HUD 姿态显示、基础运动控制下发、SQLite 框架搭建。
* **版本 v1.5 (AI Enhancement) - 第3月**
  * 核心功能：边缘侧接入 ONNX，实现红外阈值告警与自动截图，数据库证据链闭环，完善 Failsafe 机制。
* **版本 v2.0 (Swarm & SLAM) - 未来规划**
  * 核心功能：支持 1控多机台协同，引入 3D LiDAR 室内点云建图，升级为 PostgreSQL 分布式存储。
