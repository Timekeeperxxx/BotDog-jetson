# BotDog 陌生人识别与跟踪功能分阶段实现流程

## 1. 文档目标

本文档给出按当前 BotDog 仓库实际情况制定的分阶段实现路线。  
目标不是“理想化大版本规划”，而是：

1. 让团队能够按阶段直接排开发任务
2. 每阶段有明确输入、输出、改动文件、测试和验收标准
3. 每阶段结束都形成可运行、可演示、可回归的里程碑
4. 确保系统鲁棒性优先，而不是先堆功能

---

## 2. 实现难度判断

## 2.1 总体难度

### 结论
该功能整体难度评估为：**中等偏上**。

不是因为单点算法难，而是因为它横跨：

1. 视频检测
2. 多目标管理
3. 跟踪状态机
4. 控制仲裁
5. 前端状态展示
6. 事件与证据留痕
7. 安全停机边界

### 难度来源
1. 现有 AI Worker 只支持单次检测告警，不支持持续跟踪
2. 现有控制服务虽可复用，但没有自动跟踪控制权管理
3. 多目标出现时的选择与切换策略容易造成抖动
4. 自动控制与人工控制并存时需要严格优先级
5. 出区停止、丢目标停止、急停、断流停止必须全部正确

### 难度不是主要来自
1. YOLO 检测模型本身
2. WebSocket 广播本身
3. 重点区点内判定本身
4. 单条 stop 命令下发本身

真正的难点是：**模块之间的协同与状态一致性**。

---

## 2.2 技术栈判断

### 继续沿用的技术栈
- Python 3.12+
- FastAPI
- SQLAlchemy
- WebSocket
- FFmpeg
- YOLOv8
- OpenCV
- React + TypeScript
- SQLite

### 建议新增但仍属轻量范围
- 目标跟踪：可先不引入复杂 MOT 框架，第一阶段先用“单目标锁定 + 帧间保持”
- 第二阶段再接入多目标跟踪器（如 ByteTrack / BoT-SORT 的工程封装）
- 区域判断：OpenCV / 几何工具
- 前端叠加绘制：Canvas 或 SVG 即可

### 本期不建议引入
- 深度人脸识别框架
- 步态识别模型训练
- 大型消息中间件
- ROS 全量重构
- 地图导航框架

---

## 3. 分阶段总览

建议拆为 5 个阶段：

1. **阶段 0：基线修复与开发前置检查**
2. **阶段 1：单目标陌生人锁定与区内跟踪 MVP**
3. **阶段 2：多目标选择与人工打断仲裁**
4. **阶段 3：会话级已知人员标记与前端状态闭环**
5. **阶段 4：鲁棒性加固、测试补齐、联调验收**

每个阶段都应能独立运行，不依赖“后面再补”。

---

## 4. 阶段 0：基线修复与开发前置检查

## 4.1 目标
修复现有仓库中的关键不一致问题，为自动跟踪功能打基础。

## 4.2 必做项

### 4.2.1 修复 E-STOP 检查不生效
现状：
- `ControlService` 中 `_is_e_stop_active()` 读取的状态字段与 `StateMachine` 实际暴露字段不一致

结果风险：
- 急停状态下自动跟踪命令可能仍被接受

处理要求：
- 统一读取接口
- 补测试覆盖

### 4.2.2 统一 confidence 语义
现状：
- 温度告警写入的 `confidence` 与 ORM 约束不一致

处理要求：
- 统一所有事件 `confidence` 为 0~1 或明确区分 `score` / `confidence`
- 自动跟踪事件沿用统一标准

### 4.2.3 统一 Evidence Schema 空值约束
现状：
- ORM 允许部分空值
- API Schema 有字段声明不一致

处理要求：
- 自动跟踪事件接入前统一模型约束
- 避免运行时 Pydantic 校验异常

### 4.2.4 统一文档基线
处理要求：
- 以 `main.py` 当前控制接口为准
- 标记旧协议文档中过期描述

## 4.3 输出物
1. 可通过的基础测试
2. 修复后的控制服务
3. 统一后的事件字段语义
4. 开发基线说明

## 4.4 验收标准
1. 急停时 `control/command` 返回拒绝
2. stop 命令在异常状态下仍可执行
3. 证据接口无类型不一致问题

---

## 5. 阶段 1：单目标陌生人锁定与区内跟踪 MVP

## 5.1 目标
在当前项目上做出**最小可运行版本**：

- 检到人
- 满足规则判定陌生目标
- 锁定单个目标
- 在重点区内低速跟随
- 出区后停止

这是最重要的里程碑。

## 5.2 范围控制
本阶段明确只做：

1. 单目标
2. 单重点区
3. 单会话
4. 不做目标切换
5. 不做复杂 ReID
6. 不做已知人员标记
7. 不做多人决策

## 5.3 开发任务

### 5.3.1 新增跟踪状态定义
新增：
- `AutoTrackState`
- `TrackStopReason`

### 5.3.2 新增重点区判断模块
能力：
- 读取重点区 polygon
- 用目标框底部中心点判断 inside/outside

### 5.3.3 改造 AIWorker
改为：
- 不再“只要检测到人就直接告警”
- 增加“检测 -> 稳定命中 -> 陌生目标锁定 -> 跟踪控制”链路

### 5.3.4 新增 AutoTrackService
至少实现：
- `process_frame(...)`
- `enable() / disable()`
- `get_status()`
- `stop(reason)`

### 5.3.5 新增最小跟踪控制器
输出：
- `left`
- `right`
- `forward`
- `stop`

### 5.3.6 通过 ControlService 发命令
要求：
- 自动命令必须走 `ControlService`
- 不允许直接调 Adapter

### 5.3.7 事件广播
新增最小事件：
- `STRANGER_TARGET_LOCKED`
- `AUTO_TRACK_STARTED`
- `AUTO_TRACK_STOPPED`
- `AUTO_TRACK_OUT_OF_ZONE`

## 5.4 建议改动文件
- `backend/workers_ai.py`
- `backend/main.py`
- `backend/control_service.py`
- `backend/schemas.py`
- `backend/config.py`

新增：
- `backend/tracking_types.py`
- `backend/zone_service.py`
- `backend/auto_track_service.py`
- `backend/follow_decision_engine.py`

## 5.5 测试重点
1. 进入重点区后能锁定目标
2. 目标在区内时可持续输出控制命令
3. 连续出区后 stop
4. stop 后状态进入终止态
5. 非任务态无法进入自动跟踪

## 5.6 验收标准
1. 单人进入区内能自动跟踪
2. 单人出区后能自动停止
3. stop 事件被广播
4. 无人工输入时自动跟踪可持续运行

---

## 6. 阶段 2：多目标选择与人工打断仲裁

## 6.1 目标
在 MVP 基础上补上实际场景最关键的两个能力：

1. 多目标出现时只选择一个跟踪
2. 自动跟踪可被人工打断

## 6.2 开发任务

### 6.2.1 新增 TargetManager
能力：
- 维护 candidate targets
- 维护 active target
- 目标打分
- 目标保持
- 目标切换条件

### 6.2.2 接入多目标跟踪输出
本阶段开始，检测器输出不再只保留“最佳 person”，而是保留候选集合。

实现策略建议：
- 若时间紧，可先基于检测框 IOU + 连续帧匹配做简化 track id
- 若团队已有现成跟踪器封装，则此阶段接入统一 tracker

### 6.2.3 新增 ControlArbiter
能力：
- 手柄 > 网页 > 自动跟踪
- 急停最高优先级

### 6.2.4 接入人工打断状态
处理：
- 手柄输入存在时，自动跟踪立即进入 `MANUAL_OVERRIDE`
- 网页人工输入存在时，同理
- 自动发送 `stop`

### 6.2.5 新增状态与事件
状态：
- `MANUAL_OVERRIDE`
- `LOST_SHORT`

事件：
- `AUTO_TRACK_INTERRUPTED`
- `AUTO_TRACK_TARGET_SWITCHED`
- `AUTO_TRACK_TARGET_LOST`

## 6.3 建议改动文件
新增：
- `backend/target_manager.py`
- `backend/control_arbiter.py`
- `backend/stranger_policy.py`

修改：
- `backend/workers_ai.py`
- `backend/auto_track_service.py`
- `backend/main.py`

## 6.4 测试重点
1. 多人出现时只选一个目标
2. 当前目标稳定时不频繁切换
3. 新目标分数高时可切换
4. 手柄一接管，自动跟踪立刻停止
5. 网页控制一接管，自动跟踪立刻停止
6. 急停时所有自动控制立即失效

## 6.5 验收标准
1. 多人出现时画面不抖、不乱切
2. 打断动作可在 1 个控制周期内生效
3. 当前控制拥有者可查询、可展示

## 7. 阶段 3：会话级已知人员标记与前端状态闭环

## 7.1 目标
补上“不是陌生人”的可操作闭环，并让前端真正可用。

## 7.2 开发任务

### 7.2.1 新增会话级已知人员标记
能力：
- 操作员将当前 `target_id` 标记为已知
- 当前会话内不再触发陌生人跟踪
- 当前目标若正在跟踪，应立即停止

### 7.2.2 持久化会话已知标记
建议新增表：
- `session_known_targets`

### 7.2.3 新增 API
- `POST /api/v1/auto-track/mark-known`
- `GET /api/v1/auto-track/status`
- `POST /api/v1/auto-track/enable`
- `POST /api/v1/auto-track/disable`
- `POST /api/v1/auto-track/pause`
- `POST /api/v1/auto-track/resume`

### 7.2.4 前端状态面板
新增展示：
- 自动跟踪开关
- 当前状态
- 当前控制拥有者
- 当前目标 ID
- 当前停止原因
- 已知人员标记按钮

### 7.2.5 视频叠加层
可选但建议做：
- 当前目标框
- 重点区 polygon
- 当前目标高亮

## 7.3 建议改动文件
新增：
- `backend/routes_auto_track.py`（或集中注册）
- `backend/services_known_target.py`（可选）
- 前端对应 store / panel / overlay

修改：
- `backend/main.py`
- `backend/models.py`
- `backend/schemas.py`
- `backend/services_evidence.py`
- `backend/services_config.py`

## 7.4 测试重点
1. 标记 known 后目标不再被跟踪
2. 已跟踪目标被标记 known 后立即 stop
3. 自动跟踪状态接口与前端显示一致
4. 开关关闭时自动跟踪不再执行
5. 暂停 / 恢复逻辑正确

## 7.5 验收标准
1. 操作员能完成“误判纠正”
2. 前端能看到当前目标和停止原因
3. 系统状态与前端展示一致

---

## 8. 阶段 4：鲁棒性加固、测试补齐、联调验收

## 8.1 目标
把“能跑”提升为“稳定可演示、可回归”。

## 8.2 开发任务

### 8.2.1 断流与重连
- RTSP 断流时安全 stop
- FFmpeg 重连后状态可恢复到 `IDLE`
- 不自动恢复旧目标跟踪

### 8.2.2 丢目标处理加固
- `LOST_SHORT` 状态策略收敛
- 丢目标不前冲
- 丢目标结束后明确 stop reason

### 8.2.3 日志与证据规范化
- 跟踪开始 / 结束 / 打断 / 标记 known / 出区停止全部留痕
- 证据截图触发点统一

### 8.2.4 测试补齐
建议补：
- 单元测试
- 路由测试
- 自动跟踪状态机测试
- 控制仲裁测试
- 区域判断测试
- 断流降级测试
- 手动接管测试

### 8.2.5 参数调优
至少调优：
- `stable_hits`
- `target_hold_seconds`
- `out_of_zone_frames`
- `lost_timeout_frames`
- `yaw_deadband_px`
- `command_interval_ms`

## 8.3 验收脚本建议
现场联调时至少覆盖：

1. 单人进入 -> 跟踪 -> 出区停止
2. 多人进入 -> 只跟一个
3. 网页接管 -> 自动跟踪停止
4. 手柄接管 -> 自动跟踪停止
5. 急停 -> 立即停
6. 标记已知 -> 不再跟踪
7. 视频断流 -> 自动 stop
8. 任务停止 -> 自动 stop

## 8.4 验收标准
1. 演示全流程连续跑 5 次无逻辑性错误
2. 不出现自动跟踪与人工控制抢命令
3. 出区必停
4. 丢目标不乱跑
5. 所有关键事件可查证

## 11. 推荐实施顺序结论

最合理的开发顺序不是先做“已知人员白名单”或“复杂识别”，而是：

1. 先修基线不一致
2. 先让**单目标自动跟踪 + 出区停止**跑通
3. 再补**多目标选择**
4. 再补**人工打断**
5. 再补**会话级 known 标记**
6. 最后补**前端和测试闭环**

这是对当前 BotDog 仓库最省风险、最符合现有架构、也最利于短期交付的路线。
