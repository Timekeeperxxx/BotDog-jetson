# BotDog 分阶段实现方案

> **版本**：v1.0  
> **状态**：审阅稿  
> **适用范围**：仅限 `/home/jetson/Project/BOTDOG/BotDog` 仓库内  
> **硬性约束**：
> - 不修改仓库外任何文件
> - 不修改 `data/` 中现有 JSON 的格式
> - 不修改任何会影响 ROS2 话题读写的契约
> - 不推翻已确认正确的业务逻辑，只做架构、稳定性、可维护性、可发布性收敛

---

## 1. 目标

把当前系统按可控节奏推进到产品级状态，而不是一次性重构。

实施策略是分阶段推进，每个阶段都满足以下条件：

- 范围足够小，可以单独审查
- 风险可控，能回滚
- 先保稳定，再做结构，再做体验
- 不破坏 `data/` JSON 和 ROS2 契约

---

## 2. 阶段总览

### 阶段 0：准备期
- 冻结契约
- 建立基线
- 明确不可变项
- 统一验收口径

### 阶段 1：稳态保护
- 加固后端入口
- 收敛启动流程
- 保护控制、安全、导航、建图基础链路

### 阶段 2：结构收敛
- 收口配置体系
- 拆清依赖方向
- 降低全局状态使用
- 拆分前端页面级控制逻辑

### 阶段 3：产品收口
- 补齐回归测试
- 收敛日志和运维信息
- 统一错误与失败语义
- 做最终交付检查

---

## 3. 阶段 0：准备期

### 3.1 目标

建立一个不会轻易被破坏的基线，后续每一步都能回滚和验证。

### 3.2 范围

- 仅审查和确认，不改业务逻辑
- 明确哪些文件、JSON、ROS2 契约属于不可变项
- 创建或确认当前代码锚点
- 统一后续改造的审批口径

### 3.3 需要确认的内容

1. 哪些 `data/` JSON 文件属于长期契约
2. 哪些 ROS2 topic 的名称、类型、方向不可变
3. 哪些业务逻辑已经确认正确，不纳入改造
4. 哪些目录属于仓库边界内，哪些绝对不能动

### 3.4 产出

- 一份不可变项清单
- 一个可回滚的本地基线锚点
- 一份分阶段实施口径

### 3.5 验收标准

- 所有边界条件清晰
- 后续任务不需要重新解释契约范围
- 任何后续改动都能定位到具体阶段

---

## 4. 阶段 1：稳态保护

### 4.1 目标

先把系统“稳定跑起来”，避免产品化过程中出现启动不稳、控制不稳、导航不稳、建图不稳的问题。

### 4.2 范围

重点关注以下模块：

- `backend/main.py`
- `backend/config.py`
- `backend/services_config.py`
- `backend/control_service.py`
- `backend/control_arbiter.py`
- `backend/state_machine.py`
- `backend/services_mapping.py`
- `backend/services_nav_localization.py`
- `backend/api/routes/control.py`
- `backend/api/routes/nav.py`

### 4.3 实施重点

1. 入口稳定
- 让启动编排更清晰
- 让初始化顺序更明确
- 让启动失败原因更可见

2. 控制稳定
- 保持现有控制逻辑不变
- 明确安全检查、限流、watchdog、E-STOP 的责任边界

3. 导航与建图稳定
- 保持建图启动前清理导航后台进程的既有行为
- 把导航与建图的互斥关系保留为显式流程

4. 配置稳定
- 把启动时配置和可热更新配置分开看待
- 防止默认安全配置留下风险入口

### 4.4 产出

- 更清晰的启动流程
- 更稳定的控制和导航入口
- 更明确的失败日志

### 4.5 验收标准

- 后端可稳定启动
- 关键控制和导航接口行为不回退
- 建图、导航、控制之间的互斥关系不被破坏
- 核心测试仍然可重复执行

---

## 5. 阶段 2：结构收敛

### 5.1 目标

把“能用”的结构收敛成“更容易维护”的结构。

### 5.2 范围

重点关注：

- 配置层
- 服务层
- 状态层
- 前端页面层
- WebSocket 状态流

### 5.3 实施重点

1. 配置收口
- 按领域整理配置项
- 区分运行时配置与热更新配置
- 降低配置项查找成本

2. 依赖显式化
- 减少路由层直接拿全局对象
- 让服务依赖关系更明确
- 降低隐藏耦合

3. 状态收敛
- 明确谁负责维护运行态
- 避免同一状态在多个地方各存一份但语义不一致

4. 前端拆分
- 把页面级控制逻辑拆成更小业务块
- 统一加载态、错误态、空态和危险操作确认

### 5.4 产出

- 配置更易理解
- 模块边界更清楚
- 前端页面更容易维护

### 5.5 验收标准

- 配置项能按领域快速定位
- 关键服务依赖关系更清晰
- 页面复杂度下降，但行为不变

---

## 6. 阶段 3：产品收口

### 6.1 目标

把项目从“工程可运行”收敛到“可交付、可排障、可持续维护”。

### 6.2 范围

- 测试体系
- 日志体系
- 运维脚本
- 错误与失败输出
- 最终验收清单

### 6.3 实施重点

1. 测试门禁
- 补足关键链路回归测试
- 保证核心流程每次都能复现验证

2. 可观测性
- 统一启动摘要
- 统一失败摘要
- 统一关键日志字段

3. 运维收口
- 收敛脚本职责
- 明确服务启动和恢复步骤
- 减少现场排障成本

4. 最终交付
- 形成可审查的验收清单
- 每个子系统都能明确是否达标

### 6.4 产出

- 更稳定的回归保障
- 更清楚的故障定位路径
- 更适合部署和维护的系统形态

### 6.5 验收标准

- 关键测试通过
- 主要流程失败时有清晰定位信息
- 运维侧能快速判断问题属于哪一层

---

## 7. 允许修改的文件类型

在每个阶段里，优先考虑以下类别的文件：

- `backend/main.py` 和入口编排相关文件
- `backend/config.py` 和配置服务相关文件
- `backend/*service*.py` 中职责过重的模块
- `backend/api/routes/*.py` 中过度参与内部决策的路由
- `frontend/src/pages/*` 中过大的页面控制器
- `frontend/src/hooks/*` 中可拆分的状态逻辑
- `tests/*` 中关键链路测试
- `scripts/*` 中职责混杂的脚本

---

## 8. 不允许触碰的内容

- 仓库外文件
- `data/` 中现有 JSON 格式
- ROS2 topic 名称、类型、读写语义
- 已确认正确且不需要变更的业务逻辑
- 为了“看起来更干净”而引入的破坏性改动

---

## 9. 推荐执行节奏

建议按以下顺序推进：

1. 先做阶段 0，确认边界和锚点
2. 再做阶段 1，保证稳态
3. 然后做阶段 2，收敛结构
4. 最后做阶段 3，完成产品收口

每一阶段都遵守：

- 只改一个明确范围
- 改动后先做验证
- 必要时保持回滚点
- 不跨越契约边界

---

## 10. 审阅关注点

请重点审阅以下问题：

1. 阶段划分是否合理
2. 各阶段目标是否足够清晰
3. 是否存在应该提前做的项
4. 是否有不该放在某个阶段里的内容
5. 这份方案是否足够约束后续实施不越界
