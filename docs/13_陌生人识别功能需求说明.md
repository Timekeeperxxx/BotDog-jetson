# BotDog 陌生人识别与跟踪功能需求说明

## 1. 文档目标

本文档面向研发团队，定义“陌生人识别与跟踪 / 出区停止 / 手动打断 / 模式开关”功能的详细需求。设计严格建立在当前 BotDog 仓库已实现代码之上，不假设重构整套系统，不假设短期内引入完整人脸识别，仅预留未来演进接口。

本文档的目标不是产品宣讲，而是让后端、算法、前端、联调人员可以直接据此开始编码、联调和验收。

---

## 2. 现有系统基线

### 2.1 已实现能力

当前仓库已具备如下基础能力：

1. **后端主框架已完备**
   - FastAPI 应用入口、生命周期管理、静态资源挂载、REST 接口、WebSocket 接口已建立。
   - 已有 `/api/v1/system/health`、`/api/v1/session/start`、`/api/v1/session/stop`、`/api/v1/evidence`、`/api/v1/logs`、`/api/v1/control/command`、`/api/v1/control/stop`、`/api/v1/control/e-stop` 等接口。

2. **状态机已存在**
   - 已定义 `DISCONNECTED / STANDBY / IN_MISSION / E_STOP_TRIGGERED`。
   - 已支持心跳监控、任务状态更新、急停状态切换。

3. **AI Worker 已存在**
   - 已通过 FFmpeg 读 RTSP 原始帧。
   - 已支持 YOLOv8 单目标检测逻辑。
   - 已支持稳定命中、冷却、抓拍、告警广播、证据落库。

4. **事件广播链路已存在**
   - 已有 `/ws/event` 和 `EventBroadcaster`。
   - 已支持 `ALERT_RAISED` 事件推送。

5. **控制服务已存在**
   - 已有 `ControlService`、`RobotAdapter`、`Watchdog`。
   - 当前支持离散命令：`forward / backward / left / right / sit / stand / stop`。
   - 已支持速率限制、Watchdog 自动 stop。

6. **数据库已存在**
   - 已有任务表、遥测表、证据表、操作日志表、系统配置表。

7. **前端契约已存在**
   - 已有视频区、抓拍列表、状态栏、事件流契约和历史证据查询契约。

### 2.2 当前与新需求的差距

当前 AI 侧仅支持“检测到目标 -> 触发告警 -> 抓拍 -> 落库”，不支持：

1. 陌生目标的持续锁定  
2. 多目标出现时的跟踪对象选择  
3. 目标在重点区内的低速跟随  
4. 目标出区后的自动停止  
5. 自动跟踪与手柄 / 网页控制之间的仲裁  
6. 自动跟踪功能的启停开关  
7. 前端对“自动跟踪状态 / 当前目标 / 中断原因”的展示  
8. “已知人员 / 非陌生人”的会话级标记能力

## 3. 本次功能范围

### 3.1 本期必须实现

本期实现以下功能：

1. **人员检测后按安保规则判定陌生目标**
2. **陌生目标被锁定为当前跟踪目标**
3. **当目标位于重点监控区域内时，机器狗可执行低速移动跟踪**
4. **当目标连续离开重点监控区域时，停止跟踪**
5. **当画面中存在多个陌生目标时，系统按既定优先级选择一个目标进行跟踪**
6. **自动跟踪行为可被手柄或网页控制实时打断**
7. **自动跟踪功能可被开关控制，可暂停、恢复、完全关闭**
8. **事件、证据、日志、前端状态与当前系统统一**

### 3.2 本期不实现

本期明确不做以下重功能：

1. 全自主导航和路径规划
2. 高速追击
3. 长时 ReID 跨摄像头身份恢复
4. 人脸识别白名单生产级闭环
5. 步态识别深度模型训练
6. 多狗协同围堵
7. 室内 SLAM / 地图构建

### 3.3 本期预留未来扩展点

保留以下扩展能力的接口位：

1. 人脸识别身份服务接口
2. 人体特征 / ReID 特征服务接口
3. 连续速度控制接口（替换当前离散命令）
4. B2 专用适配器接口
5. 长期人员白名单 / 授权清单

## 4. 术语定义

### 4.1 陌生人
在本期实现中，“陌生人”不是指完成了身份证明级别的身份认证，而是指：

**满足“人员检测结果”且满足安保规则触发条件的目标。**

本期安保规则优先采用：

1. 巡检任务执行中
2. 人进入重点监控区域
3. 当前未被人工标记为已知人员
4. 当前未被系统忽略
5. 满足稳定命中阈值

### 4.2 已知人员
本期的“已知人员”只支持以下两类：

1. **会话级人工标记已知**
   - 操作员在当前任务会话中将某个 `target_id` 标记为已知
   - 仅对当前任务 / 当前会话有效

2. **规则级已授权人员**
   - 非通过视觉身份识别，而是通过任务配置指定：
     - 某时段允许人员进入
     - 某区域允许维护人员存在
   - 本质是“规则放行”，不是生物识别

### 4.3 自动跟踪
本期的“自动跟踪”定义为：

**机器狗依据视觉检测结果对当前目标执行朝向调整和低速移动，使目标尽量保持在视野中心，并维持有限距离。**

### 4.4 出区停止
指当前目标锚点连续若干帧位于重点监控区域外时，自动停止一切自动跟踪移动命令。

### 4.5 打断
指自动跟踪被更高优先级控制源终止或暂停，包括：

1. 手柄操控接管
2. 网页人工控制接管
3. 急停
4. 自动跟踪开关关闭
5. 当前任务停止
6. 目标丢失
7. 目标出区

## 5. 角色与使用场景

### 5.1 角色

#### 5.1.1 操作员
通过网页或手柄监控和控制系统，能启动 / 停止自动跟踪、标记已知人员、手动接管机器狗。

#### 5.1.2 安保审核员
查看告警记录、证据截图、跟踪开始与终止原因。

#### 5.1.3 研发运维人员
负责配置重点区、调节阈值、排查事件链路问题。

### 5.2 使用场景

#### 场景 A：单人进入重点区
- 机器狗巡检中
- 一人进入重点监控区
- 系统锁定该目标并进行低速跟随
- 目标离开重点区后停止跟随

#### 场景 B：多人同时进入重点区
- 画面中有多个陌生目标
- 系统根据优先级选择一个目标进行自动跟踪
- 其余目标只做检测与告警展示，不做同时跟随

#### 场景 C：自动跟踪过程中操作员接管
- 自动跟踪执行中
- 操作员通过手柄或网页发送控制命令
- 自动跟踪立即让位
- 系统记录“manual_override”事件

#### 场景 D：误判人员被标记为已知
- 系统将某目标判定为陌生人
- 操作员确认其为授权人员
- 在当前会话中将该目标标记为已知
- 后续不再触发陌生人跟踪

## 6. 功能需求

## 6.1 总体功能链

系统完整业务链路如下：

1. 巡检任务开始
2. AI Worker 持续获取视频帧
3. 检测到人员目标
4. 根据规则判断是否为陌生目标
5. 若是，则锁定当前 `target_id`
6. 若目标在重点区内，则进入自动跟踪
7. 若出现更高优先级目标，可切换跟踪对象
8. 若目标离开重点区，则停止跟踪
9. 若人工接管或急停，则立即中断自动跟踪
10. 写入事件、证据、日志并广播到前端

---

## 6.2 功能需求明细

### FR-001 巡检任务前置约束
自动跟踪功能仅在 `IN_MISSION` 状态下允许工作。

#### 输入
- 系统状态机状态
- 当前任务是否存在

#### 规则
- 非 `IN_MISSION` 时，自动跟踪不得进入执行态
- 若任务停止，自动跟踪立即退出并下发 `stop`

#### 输出
- 自动跟踪状态变为 `DISABLED` 或 `IDLE`

---

### FR-002 人员检测
系统应持续从视频帧中检测人员目标。

#### 输入
- RTSP 视频帧

#### 处理
- 使用现有 YOLO 检测能力
- 本期检测类别只要求 `person`

#### 输出
- 检测框列表
- 每个检测目标的置信度
- 原始图像帧索引 / 时间戳

---

### FR-003 陌生人判定
系统应基于短期规则而非强身份识别判定是否为陌生目标。

#### 判定规则
满足以下全部条件时判定为陌生目标：

1. 类别为 `person`
2. 当前处于巡检任务中
3. 检测框锚点位于重点监控区域内
4. 连续命中达到稳定阈值
5. 该目标未被人工标记为已知
6. 该目标未被列入当前会话忽略列表

#### 说明
- 本期不依赖人脸识别
- 本期不承诺“生物学意义上的陌生人识别”
- 本期是“安保规则上的陌生目标识别”

---

### FR-004 目标锁定
系统一旦判定陌生目标，应创建当前跟踪对象。

#### 输出字段
- `target_id`
- `first_seen_at`
- `last_seen_at`
- `current_bbox`
- `inside_zone`
- `track_state`
- `selection_score`
- `is_known_person`
- `manual_ignored`

#### 约束
- 任一时刻只允许一个“当前自动跟踪目标”
- 可存在多个候选目标，但只能有一个 active target

---

### FR-005 多目标选择
当画面中有多个陌生目标时，系统必须按优先级选择一个目标进行自动跟踪。

#### 默认优先级规则
按以下顺序排序：

1. 当前已锁定目标优先保持不切换
2. 位于重点区核心区域的目标优先
3. 距离机器狗更近的目标优先
4. 目标框面积更大的优先
5. 置信度更高的优先
6. 首次进入区域时间更早的优先

#### 说明
- 不允许频繁来回切换目标
- 必须设置最短保持时间 `target_hold_seconds`
- 仅当新目标分数高出阈值才允许抢占

---

### FR-006 自动跟踪行为
当当前目标处于重点区内，系统应驱动机器狗执行低速跟踪。

#### 行为定义
自动跟踪由以下动作组成：

1. 机身转向，使目标尽量靠近画面中心
2. 在目标距离较远时低速前进
3. 在目标距离合适时停止前进，仅维持朝向
4. 不允许高速追击
5. 不允许后退跟随作为默认行为
6. 丢失目标时不允许盲目前冲

#### 本期控制方式
- 先基于现有离散控制命令实现：
  - `left`
  - `right`
  - `forward`
  - `stop`

#### 未来扩展
- 可替换为速度控制：
  - `vx`
  - `yaw_rate`

---

### FR-007 出区停止
当目标连续若干帧位于重点区外，系统必须停止自动跟踪。

#### 判定方式
- 使用目标框底部中心点作为区域锚点
- 使用重点区多边形判断目标是否在区内

#### 终止条件
- 连续 `N_OUT` 帧区外，或者
- 持续 `T_OUT_CONFIRM` 时间区外

#### 执行动作
1. 发送 `stop`
2. 自动跟踪状态改为 `STOPPED_OUT_OF_ZONE`
3. 广播事件
4. 记录日志
5. 可选抓拍一张终止截图

---

### FR-008 目标丢失处理
若目标短时消失，系统应进入短时丢失状态，而不是立即结束。

#### 规则
- 短时丢失：进入 `LOST_SHORT`
- 丢失期间只允许：
  - 原地轻微转向搜索
  - 或保持停止
- 丢失超时后，终止跟踪

#### 终止动作
- 发送 `stop`
- 广播 `TARGET_LOST`
- 记录日志和事件原因

---

### FR-009 手柄 / 网页打断
自动跟踪必须可被手柄或网页控制实时打断。

#### 控制优先级
从高到低：

1. 急停
2. 手柄人工控制
3. 网页人工控制
4. 自动跟踪

#### 行为规则
- 一旦检测到人工控制输入，自动跟踪立即停发控制命令
- 当前状态切换为 `MANUAL_OVERRIDE`
- 记录打断来源和时间
- 人工释放后，不自动恢复到原目标，除非显式点击“恢复自动跟踪”

---

### FR-010 自动跟踪开关
系统必须支持自动跟踪的启停开关。

#### 开关形式
1. 后端全局配置开关
2. 前端页面开关
3. 当前任务级开关

#### 开关效果
- 关闭后不再进入自动跟踪流程
- 已经在跟踪中的目标立即停止并下发 `stop`
- 检测与告警可根据配置决定是否继续保留

---

### FR-011 已知人员标记
系统应允许操作员在当前会话中标记“这个人不是陌生人”。

#### 本期实现要求
- 只支持**会话级别标记**
- 标记对象是当前 `target_id`
- 当前任务结束后该标记失效

#### 标记后的行为
- 不再对该 `target_id` 触发陌生人跟踪
- 已触发的当前跟踪立即结束
- 记录“mark_known_person”事件

---

### FR-012 事件广播
系统必须将自动跟踪相关状态广播给前端。

#### 必须新增的事件类型
- `STRANGER_TARGET_LOCKED`
- `AUTO_TRACK_STARTED`
- `AUTO_TRACK_STOPPED`
- `AUTO_TRACK_INTERRUPTED`
- `AUTO_TRACK_TARGET_SWITCHED`
- `AUTO_TRACK_OUT_OF_ZONE`
- `AUTO_TRACK_TARGET_LOST`
- `AUTO_TRACK_MARKED_KNOWN`

---

### FR-013 证据与日志
系统必须对关键状态变更做留痕。

#### 必须留痕的节点
1. 陌生目标首次锁定
2. 自动跟踪开始
3. 自动跟踪结束
4. 结束原因
5. 被人工打断
6. 被标记为已知
7. 目标丢失
8. 出区停止

---

### FR-014 前端展示
前端必须可视化展示自动跟踪状态。

#### 必须展示
1. 自动跟踪开关状态
2. 当前自动跟踪目标 ID
3. 当前跟踪状态
4. 当前目标是否在重点区内
5. 当前目标被中断的原因
6. 当前是否被人工接管
7. 最近一次跟踪事件

#### 可选增强
1. 视频叠加目标框
2. 视频叠加重点区多边形
3. 当前目标标识高亮

---

## 7. 非功能需求

### 7.1 实时性
1. 人员检测到自动跟踪启动的决策延迟应尽可能控制在 500ms 内
2. 打断指令应优先于自动跟踪执行
3. `stop` 命令必须可以绕过普通限流

### 7.2 鲁棒性
1. RTSP 断流时自动跟踪必须自动降级为停止态
2. 前端断开不影响自动跟踪主流程
3. EventBroadcaster 无连接时，不得阻塞主流程
4. 数据库存储失败时，控制与安全逻辑不得受影响

### 7.3 安全性
1. 急停最高优先级
2. 丢失目标后禁止前冲
3. 出区必须停止
4. 非任务态禁止自动跟踪
5. 系统关闭自动跟踪开关后必须立即停机身动作

### 7.4 可维护性
1. 检测、目标管理、控制决策、控制仲裁、区域判断必须拆分模块
2. 新状态机不得硬编码在单文件大函数中
3. 所有新增状态均需前后端统一定义

## 8. 现有代码基线修复前置项

在正式开发本功能前，必须先修复以下已存在的不一致，否则自动跟踪的鲁棒性无法保障。

### 8.1 E-STOP 检查字段不一致
当前控制服务的急停检查逻辑使用了不存在的 `current_state`，而状态机实际暴露的是 `state`。  
该问题必须先修复，否则 E-STOP 无法可靠阻断自动跟踪控制。

### 8.2 温度告警 confidence 语义不一致
当前温度告警将 `confidence` 按百分数写入，但现有 ORM 约束要求 `confidence` 在 0~1。  
新增自动跟踪事件时必须统一 `confidence` 语义。

### 8.3 Evidence Schema 空值约束不一致
当前证据模型允许部分字段为空，但 API Schema 某些字段为非空。  
新增自动跟踪事件证据前，应统一 Schema 约束。

### 8.4 文档与实现不一致
旧协议文档仍写“Web 控制已下线”，但当前后端已有网页控制接口。  
新功能文档、实现、前端契约必须以**当前代码**为准，不再沿用旧表述。

---

## 9. 验收标准

满足以下条件视为本功能完成：

1. 单人进入重点区时，系统能锁定并开始低速自动跟踪
2. 多人同时出现时，系统能稳定选择一个目标，不发生高频切换
3. 目标连续出区时，系统自动停止跟踪
4. 目标丢失后，系统不会盲目前冲
5. 手柄控制可立即打断自动跟踪
6. 网页控制可立即打断自动跟踪
7. 自动跟踪开关关闭后，自动跟踪立即停止
8. 操作员可将当前目标标记为已知人员
9. 前端可看到当前跟踪状态和停止原因
10. 关键事件均可查询、可回放、可留痕

## 10. 未来扩展建议

1. 将“已知人员”从会话级标记升级为长期授权档案
2. 引入人脸识别服务作为可插拔身份源
3. 引入人体 ReID / 步态特征服务
4. 将离散指令跟踪升级为连续速度控制
5. 接入 B2 专用适配器，支持更平滑的跟踪控制
