# 一、你的需求是什么

你的目标不是“自主导航巡逻”，也不是“追着人跑”，而是一个​**固定场景下的视觉触发式防区驱离闭环**​。

更准确地说，你要的是：

机器狗在固定起始观察位待命，通过机载摄像头持续观察前方地面上的​**要害区域**​。这个要害区域不再靠人工画死一个静态 polygon，而是由现场一块**黄色区域 + 黑色边框**的物理标靶来表示。系统需要持续识别这块区域在当前画面中的真实位置和形状。

当 YOLO 检测到 `person`，并且人的**脚点**进入该区域后，系统判定有人进入要害区域，触发驱离。触发后，机器狗不去盯着人追，而是始终以这块区域本身作为视觉参考，朝区域方向推进，并通过喇叭进行警示。

当人离开该区域并持续一段时间后，系统判定驱离成功。随后机器狗不是按固定秒数盲退，而是继续利用这块区域在画面中的位置、面积和形状变化做​**视觉返航**​，尽量回到原来的观察位，最后停止并恢复待命姿态。

同时，这套方案必须考虑一个现实问题：​**人站上区域后会遮挡区域**​。所以系统不能在遮挡状态下把区域错误切换到别的相似颜色块上，而必须具备“冻结最近一次稳定区域、短时容错、严重遮挡则保护停机”的能力。

# 二、你的核心技术路线

你的路线现在已经比较清楚了，本质上是下面这套分工：

## 1. YOLO 只负责识别人

YOLO 的职责只有一个：检测 `person`，给出 bbox。
后续只取 **脚点** 参与“是否进入区域”的判定，不拿整个人框直接压区域。

## 2. OpenCV 负责识别要害区域

要害区域不是用普通 YOLO 去识别“黄色”，而是用传统视觉方法去做：

* 黄色区域分割
* 黑色边框验证
* 轮廓提取
* 四边形恢复
* 输出动态 polygon / bbox / center / area / angle

也就是说，你识别的不是“汉字”“木板”“某个物体类别”，而是：

**地面上那块黄色主体 + 黑色边框构成的区域几何结构。**

## 3. 驱离和返航都以“区域”为视觉锚点

前进时不是看人往哪跑，而是看区域在画面里的：

* 中心偏左还是偏右
* 面积是变大还是变小
* 是否已经接近“完整看到区域”的状态

返航时则看当前区域和起始区域之间的差异：

* 中心是否回到起始附近
* 面积是否回到起始面积附近
* 四边形形状是否接近起始状态

所以这条路线本质上是：

**YOLO 识别人，区域检测模块识别防区，GuardMission 用区域做视觉伺服。**

---

# 三、你真正要解决的难点

你这个项目的难点不在“怎么检测到人”，而在下面三件事。

## 1. 区域要动态识别，不是静态框

区域不能再是起始时刻的静态 bbox。
因为狗一动，视角就变，区域在画面中的位置、面积和形状都会变。

## 2. 人踩上去后区域会被遮挡

所以系统不能每一帧都从零开始在全图找“哪里最黄”，否则很容易跳到别的相近颜色区域。

## 3. 返航不能再靠固定秒数

否则误差会累计，越跑越偏。
必须让返航和当前画面中的区域状态绑定。

你现在的 **BotDog-jetson** 已经把 `GuardMissionService` 接进主流程了，`AIWorker` 在启用 guard 模式时也会优先把检测结果交给它；所以现在不需要推翻总架构，真正要改的是 ​**GuardMissionService 内部逻辑**​，把它从“静态框 + 定时前后退”改成“动态区域检测 + 视觉伺服”。

而你当前代码里的问题是：虽然 `GuardMissionService`、`visual_anchor_tracker`、`visual_servo_controller` 都已经进仓，但 `guard_mission_service.py` 现在实际上仍然主要依赖**静态防区外接框**和​**固定时长推进/返航**​，并没有真正形成动态区域闭环。

所以最合理的落地路径是下面这条。

# 五、具体实现路径

## 阶段 1：把物理区域做对

先不要急着改代码，先把现场目标区做成适合识别的样子。

推荐做法：

* 中间是**纯黄色哑光面**
* 四周是一圈**粗黑边**
* 尽量是规则长方形
* 不要反光，不要木纹，不要杂色
* 条件允许的话，四角再加简单角标

这一步的目的不是好看，而是让区域在复杂背景里尽量“唯一”。

---

## 阶段 2：新增动态区域检测模块

新增一个模块，例如：

`backend/yellow_zone_detector.py`

它每帧输入图像，输出：

* `polygon`
* `bbox`
* `center`
* `area`
* `angle`
* `quality/confidence`

### 它内部怎么做

核心流程就是：

1. BGR 转 HSV
2. 阈值分割黄色区域
3. 形态学去噪
4. 找轮廓
5. 四边形拟合
6. 黑边验证
7. 输出最可信的目标区

注意，这里识别的不是“矩形”，而是​**图像中的四边形投影**​。
因为地面上的矩形区域在相机视角下，通常表现为四边形，而不是严格矩形。

---

## 阶段 3：把“入侵判定”改成脚点进 polygon

当前 GuardMission 里的入侵判断偏粗，你现在真正应该做的是：

* YOLO 检测 `person`
* 取 bbox 底边中点作为脚点
* 判断脚点是否落在当前动态区域 polygon 内
* 连续持续若干帧后确认入侵

这样比“整个人框和区域框有交集”稳很多，也更符合“人踩进区域”的物理语义。

---

## 阶段 4：加入遮挡容错，不允许区域跳到别处

这是整个方案能不能稳定的关键。

你必须维护两个区域状态：

* `last_good_zone`：最近一次高质量、无遮挡或轻遮挡时的稳定区域
* `current_zone`：当前帧检测结果

### 更新规则

如果当前帧质量高，就更新 `last_good_zone`。
如果当前帧质量下降，但还没彻底丢，就优先沿用 `last_good_zone`，只允许很小的修正。
如果连续多帧质量都很差，就进入 `LOST_ZONE` 或 `LOST_ANCHOR`，停止继续推进。

### 这一步的本质

不是“继续找别的黄色块替代”，而是：

**遮挡时优先相信历史稳定区域，不轻易换目标。**

---

## 阶段 5：把前进改成真正的视觉伺服

当前 `guard_mission_service.py` 里的 `_on_advancing()` 需要重写。

前进阶段每帧应该做：

1. 调 `yellow_zone_detector.detect()`
2. 得到当前动态区域
3. 用 `curr_zone.center` 和画面中心比较，决定 `left/right`
4. 用 `curr_zone.area` 判断是否已经靠近
5. 只要人脚点还在区域内，就继续推进
6. 人脚点离开区域并持续若干秒，就进入返航

### 你说的“走到最后能完整看到整个区域为止”

这可以直接变成一个保护条件：

* 当前区域面积 / 画面面积 达到某个阈值
* 或区域四边形已经接近视野边界
* 就不再继续推进，只保持警示

---

## 阶段 6：把返航改成真正的视觉回归

当前 `_on_returning()` 不能再按固定时间后退。

你应该在入侵刚触发时记录：

* `start_polygon`
* `start_center`
* `start_area`
* `start_angle`

返航阶段每帧动态识别区域，然后比较：

* `curr_center` vs `start_center`
* `curr_area` vs `start_area`
* `curr_angle` vs `start_angle`

然后调用 `visual_servo_controller.compute_returning(...)` 去产生：

* `left`
* `right`
* `backward`
* `stop`

### 返航完成条件

不是某一帧满足就停，而是：

* 位置误差小于阈值
* 面积误差小于阈值
* 角度误差小于阈值
* 连续 N 帧都成立

然后：

* `stop`
* `sit`
* 回到 `STANDBY`

---

## 阶段 7：让 `visual_servo_controller` 真正参与主控制

你现在这个模块已经有雏形，但主流程没用好。
接下来要做的是：

### `compute_advancing(...)`

输入：

* `curr_bbox / curr_polygon`
* `frame_width / frame_height`
* `max_view_ratio`

输出：

* `cmd`
* `is_arrived_edge`

### `compute_returning(...)`

输入：

* `curr_bbox`
* `start_bbox`
* `frame_width`
* 容差参数

输出：

* `cmd`
* `is_returned`

也就是说，不要再在 `GuardMissionService` 里手写一套粗糙的 `if left/right/forward`，而是让 servo controller 变成真正的决策核心。

# 六、推荐的状态机

结合你的最终需求，最合适的状态机其实可以简化成：

## `STANDBY`

* 持续识别动态区域
* 持续识别人
* 人脚点进入区域并持续确认 → `ADVANCING`

## `ADVANCING`

* 以动态区域为视觉参考推进
* 人仍在区域内 → 继续
* 人离开区域并持续若干秒 → `RETURNING`
* 区域到达视场上限 → 停止推进但继续警示
* 区域丢失超时 → `LOST_ZONE`

## `RETURNING`

* 以动态区域为视觉参考返航
* 当前区域回到起始区域附近并持续稳定 → `STANDBY`

## `LOST_ZONE`

* 立即停机保护
* 等待人工处理或自动恢复

# 七、这条路线的优点

这条路线最适合你现在的原因是：

它直接利用你场地里那块“黄色地板 + 黑色边框”做视觉锚点。
它比你现在“静态 bbox + 定时前后退”稳得多。
同时，它和你当前项目已经接好的 `GuardMissionService` 主干是兼容的，不用推倒重来。

# 八、一句话总结

你的需求可以概括为：

**让机器狗在固定观察位通过机载视觉监控一块黄色+黑边的地面要害区域，检测到人脚进入该区域后启动驱离，驱离过程中始终以该区域本身作为视觉参考推进，待人离开区域后再依据该区域在画面中的位置和面积变化完成视觉返航，并在区域被遮挡时通过历史稳定区域冻结和丢失保护避免跳到其他相似颜色块上。**
