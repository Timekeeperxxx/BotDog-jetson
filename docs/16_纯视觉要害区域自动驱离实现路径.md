# BotDog 纯视觉要害区域自动驱离实现路径

## 1. 目标定义

在现有 BotDog 系统基础上，实现如下完整闭环：

1. 机器狗在起点位置待命（蹲坐姿态）；
2. 机载摄像头在起点视角下，持续监测配置的重点区域（像素 polygon）；
3. 当 YOLO 检测到目标（person）进入重点区域并持续确认后，系统判定入侵成立；
4. 机器狗起立，按预设运动脚本前往驱离点（固定距离、固定方向）；
5. 在驱离点通过外接音频播放器播放警告音进行驱离，同时持续检测视野中是否还有 person；
6. 当视野中连续若干秒无 person 后，系统判定目标已驱离；
7. 机器狗按预设运动脚本返回起点，蹲坐恢复待命姿态。

**核心定位：这不是自主导航问题，而是"固定场景、固定路径、视觉触发"的任务闭环问题。**

---

## 2. 设计原则

### 2.1 不做什么（明确排除）

| 排除项 | 原因 |
|--------|------|
| 地面坐标系建模 | 当前场景固定、路径固定，不需要地面真实坐标 |
| 视觉标记/ArUco 定位 | 增加外部依赖和标定成本，MVP 阶段不需要 |
| 视觉闭环回位 | B2 速度级控制的开环重复性在短距离（<5m）已足够 |
| SLAM / 里程计 | B2 SDK 不提供里程计反馈，引入外部传感器超出范围 |
| 修改现有 AutoTrackService | 它的职责是"跟踪目标"，与"定点驱离"是两个独立功能 |

### 2.2 要做什么（核心策略）

1. **分阶段检测策略**（详见 2.3）：起点用像素 polygon 判断入侵，驱离点用全画面 person 检测判断清空；
2. **新增独立的驱离任务服务** `GuardMissionService`：不塞进 AutoTrackService；
3. **用参数化开环运动脚本**实现前往和返回：距离短（1~5m），速度低（0.3m/s），开环误差可接受；
4. **前往和返回的运动时长独立可调**：真机上前进和后退的实际速度、稳定性、打滑程度不一定对称；
5. **保持与现有架构一致**：经过 ControlService -> ControlArbiter -> RobotAdapter 的标准链路；
6. **每个阶段独立可验证**，不需要等全部完成才能测试。

### 2.3 机载摄像头视角问题与分阶段检测策略

**问题**：摄像头装在机器狗身上，当机器狗从起点移动到驱离点后，视角完全变化，原先在起点配置的像素 polygon 不再对应同一块真实地面区域。因此不能在所有状态下统一使用同一套区域判断逻辑。

**解决方案：按任务阶段切换检测策略**

| 任务阶段 | 检测策略 | 原因 |
|----------|---------|------|
| **STANDBY**（起点待命） | ZoneService 像素 polygon 判断 | 起点视角固定，polygon 有效 |
| **DEPLOYING**（前往驱离点） | 不做检测 | 运动脚本驱动，不需要视觉反馈 |
| **GUARDING**（驱离中） | **全画面 person 检测**（不用 polygon） | 视角已变化，polygon 失效；但因驱离点面朝要害区域方向，视野中的 person 就是需要驱离的目标 |
| **RETURNING**（返回起点） | 不做检测，不允许重新触发 | 运动脚本驱动；即使再次检测到人也先回 STANDBY，下一轮再触发 |

**为什么全画面检测在驱离点可行**：
- 驱离点的位置和朝向是预设的，面朝要害区域方向
- 在驱离点视野中出现的 person，大概率就在要害区域附近
- 驱离的目的是"赶走人"，不是精确判断人在哪个区域边界上
- 因此"视野中连续无 person"在语义上约等于"要害区域已清空"

**GUARDING 阶段清空判定的语义边界**：

MVP 阶段的"清空"语义不是"真实要害区域严格几何清空"，而是**"驱离点当前视野方向内持续无 person，视为处置完成"**。这是工程近似，存在两类已知误判风险：

- **漏判清空**：目标已离开要害区域，但画面边缘仍有无关人员经过，系统误认为"仍有入侵者"
- **误判清空**：目标暂时离开镜头视野或被遮挡，但并未真正离开要害区域方向

为降低边缘误检干扰，GUARDING 阶段用于清空判定的 person 检测应满足**最小置信度门槛**和**最小检测框尺寸门槛**，过滤远处小人影、半截身体、偶发误检等低质量结果。

如果上机验证后发现全画面检测的误判率影响实用性，**第一优先升级项**是引入"驱离点专用 polygon"（预先标定驱离点视角下的一套独立 polygon），将全画面检测收窄为驱离方向的定向检测。

**方案成立的前提条件**：

1. 机器狗在 STANDBY 蹲坐姿态下，机载摄像头能够稳定覆盖重点区域；否则入侵触发链条不成立。
2. 机器狗在驱离点站立姿态下，机载摄像头能够稳定朝向需要驱离的方向；否则清空判定不可靠。
3. 起点和驱离点之间的路径是平坦无障碍的直线；否则开环运动脚本不适用。

## 4. 状态机设计

### 4.1 状态定义

```
GuardMissionState（6 态）:

  STANDBY          ──入侵确认──> DEPLOYING
  DEPLOYING        ──脚本完成──> GUARDING
  GUARDING         ──区域清空──> RETURNING
  RETURNING        ──脚本完成──> STANDBY（期间不检测，不重新触发）
  MANUAL_OVERRIDE  ──释放控制──> STANDBY（当前任务作废，不恢复中间状态）
  FAULT            ──人工重置──> STANDBY
```

| 状态 | 含义 | 机器狗姿态 |
|------|------|-----------|
| `STANDBY` | 起点待命，蹲坐监控 | sit（蹲坐） |
| `DEPLOYING` | 起立并前往驱离点 | stand -> forward |
| `GUARDING` | 驱离点播放警告音驱离 | stand（站立不动） |
| `RETURNING` | 返回起点（不检测、不重新出动） | backward |
| `MANUAL_OVERRIDE` | 人工接管，当前任务已中止 | 保持当前 |
| `FAULT` | 异常（视频断流等） | stop + 保持 |

### 4.2 状态转换条件

```
STANDBY -> DEPLOYING:
  前提（系统就绪条件，全部满足才允许触发）：
  - 任务开关 enabled == True
  - 控制服务已就绪（B2 适配器已初始化）
  - 当前不处于 E_STOP
  - 当前没有人工接管（ControlArbiter 未被抢占）
  - 距上次任务完成已过冷却时间
  触发条件：
  - 起点视角下，ZoneService polygon 内连续检测到 person 达到入侵确认时间（默认 1.5s）

DEPLOYING -> GUARDING:
  - 前往运动脚本执行完成（由 deploy_duration_s 控制）

GUARDING -> RETURNING:
  - 驱离点视野内连续无满足门槛的 person 达到清空确认时间（默认 5s）
    【注：此时不用 polygon，因为视角已变化；详见 2.3 清空语义边界说明】
    【注：person 需满足最小置信度和最小框尺寸门槛，过滤低质量检测】
  - 且驱离已持续 >= 最短驱离时间（默认 10s）

RETURNING -> STANDBY:
  - 返回运动脚本执行完成 + sit 动作完成
  - RETURNING 期间即使再次检测到人，也不重新出动，先完成回位再说

任意 -> MANUAL_OVERRIDE:
  - ControlArbiter 控制权被 WEB_MANUAL 或 REMOTE_CONTROLLER 抢占
  - 当前驱离任务立即终止：运动脚本取消、音频停止、机器狗 stop
  - 不尝试从中间状态恢复，不做"从 DEPLOYING 恢复"或"从 RETURNING 恢复"

MANUAL_OVERRIDE -> STANDBY:
  - 操作员释放人工控制后，系统回到 STANDBY 重新待命
  - 本次驱离任务视为作废
  - 实现要求：必须完整清零任务上下文（见 _reset_mission_context）

任意 -> FAULT:
  - 视频流断开或视觉链路失效（连续超过阈值时间无新帧）
  - 运动命令连续被拒绝（ControlService 返回非 ACCEPTED）
  - E_STOP 触发
```

### 4.3 入侵确认与清空判定的消抖设计

业务语义以**时间**为基准，实现时按 AI 实际处理帧率折算为对应帧数。当 AIWorker 的帧率、skip 参数或 suspect 模式发生变化时，等效时间自动调整。

```
intrusion_counter（STANDBY 阶段，使用 ZoneService polygon）:
  业务语义：入侵确认时间 1.5s
  实现：按 AI 实际处理帧率折算为帧数（如 10fps 实际处理 -> 15 帧）
  - 每帧在重点区域 polygon 内检测到 person -> counter++
  - 每帧未检到 -> counter = max(0, counter - 2)  (快降慢升)
  - counter >= 折算帧数 -> 确认入侵

clear_counter（GUARDING 阶段，使用全画面 person 检测）:
  业务语义：清空确认时间 5s
  实现：按 AI 实际处理帧率折算为帧数
  过滤条件：仅统计满足最小置信度（默认 0.4）和最小框面积（默认 2000px）的 person
  - 驱离状态中，全画面无满足门槛的 person -> counter++
  - 全画面有满足门槛的 person -> counter = 0  (有一帧就重置)
  - counter >= 折算帧数 -> 确认清空

注意：两个 counter 使用不同的检测策略，入侵判断依赖像素区域，
      清空判断依赖全画面检测，因为两个阶段的摄像头视角不同。

GUARDING 期间，机器狗保持站立不动播放警告音，摄像头画面稳定，
不存在运动模糊问题，因此可以持续进行清空判定。
```

## 5. 运动脚本设计

### 5.1 核心思路

B2 的运动接口是速度级的 `Move(vx, vy, vyaw)`，没有位置反馈。
我们用 **"以固定速度运动固定时间"** 的方式实现定点移动。

### 5.2 前往驱离点（DEPLOYING）

```python
deploy_script = [
    ("stand",    settle_s),       # 起立，等稳定
    ("forward",  deploy_duration), # 前进 deploy_duration 秒
    ("stop",     0.5),             # 停止，等 0.5s 稳定
]
```

**参数**：
- `GUARD_DEPLOY_DURATION_S`: 前往驱离点的前进时间（秒），默认 10.0s
- 运动速度复用 `UnitreeB2Adapter` 的 `vx`（默认 0.3 m/s）
- 建议初始值 = 目标距离 / vx，上机后根据实际效果微调

### 5.3 返回起点（RETURNING）

```python
return_script = [
    ("backward", return_duration), # 后退 return_duration 秒
    ("stop",     0.5),             # 停止
    ("sit",      settle_s),        # 蹲下，等稳定
]
```

**说明**：
- `GUARD_RETURN_DURATION_S`: 返回起点的后退时间（秒），默认 10.0s
- 前往和返回的时长**独立配置，不强行对称**，因为真机上前进和后退的实际速度、稳定性、打滑程度不一定一样
- 直线后退回到起点，不需要转身（保持面向要害区域）
- 在短距离、低速、平整地面条件下，单次往返后回位误差应控制在可接受范围内
- 多次循环累计偏移允许人工校正
- 后续版本可升级为加入里程计闭环

### 5.4 运动脚本执行器

```python
class MotionScriptRunner:
    """按顺序执行一系列 (command, duration) 运动指令。"""

    def __init__(self, watchdog_timeout_s: float = 0.5):
        # 续命间隔由 Watchdog 超时时间推导，取超时的 1/2 ~ 1/3
        # 不在设计层写死固定值，确保与 ControlService 配置一致
        self._keepalive_interval = watchdog_timeout_s / 2
        self._max_reject_count = 3  # 连续被拒绝次数上限

    async def run(self, script: list[tuple[str, float]], 
                  control_service: ControlService,
                  cancel_event: asyncio.Event) -> bool:
        reject_count = 0
        for cmd, duration in script:
            if cancel_event.is_set():
                await control_service.handle_command("stop")
                return False
            ack = await control_service.handle_command(cmd)
            # 检查命令是否被接受
            if ack.result != "ACCEPTED":
                reject_count += 1
                if reject_count >= self._max_reject_count:
                    await control_service.handle_command("stop")
                    return False  # 连续被拒绝，脚本执行失败
            else:
                reject_count = 0
            # 持续运动命令需要周期性续命（间隔 < Watchdog 超时）
            if cmd in ("forward", "backward", "left", "right"):
                elapsed = 0.0
                while elapsed < duration:
                    if cancel_event.is_set():
                        await control_service.handle_command("stop")
                        return False
                    await asyncio.sleep(self._keepalive_interval)
                    elapsed += self._keepalive_interval
                    ack = await control_service.handle_command(cmd)
                    if ack.result != "ACCEPTED":
                        reject_count += 1
                        if reject_count >= self._max_reject_count:
                            await control_service.handle_command("stop")
                            return False
                    else:
                        reject_count = 0
                await control_service.handle_command("stop")
            else:
                # 一次性命令（stand/sit/stop），等待 duration 秒
                await asyncio.sleep(duration)
        return True
```

**设计要求**：脚本执行器需要感知控制命令应答结果（`ControlAckDTO.result`）；连续被拒绝（如 `REJECTED_E_STOP`）或连续限流时，应视为脚本执行失败，调用方据此进入 FAULT 状态。

### 阶段 3：GuardMissionService 核心状态机（1 天）

**目标**：实现完整 6 态状态机和帧处理逻辑。

**新增文件**：`guard_mission_service.py`

**核心逻辑**：

```python
class GuardMissionService:
    """驱离任务主服务。"""

    def __init__(self, *, zone_service, control_service, 
                 control_arbiter, event_broadcaster, config):
        self._state = GuardMissionState.STANDBY
        self._intrusion_counter = 0
        self._clear_counter = 0
        self._guard_start_time = 0.0
        self._last_mission_end_time = 0.0
        self._last_frame_time = 0.0  # 视觉链路健康检测
        self._script_runner = MotionScriptRunner(
            watchdog_timeout_s=config.CONTROL_WATCHDOG_TIMEOUT_MS / 1000.0
        )
        self._cancel_event = asyncio.Event()
        self._script_task: Optional[asyncio.Task] = None
        # 业务时间 -> 帧数折算
        # 注意：不直接用 config.AI_FPS（那是采集帧率），而是用 AIWorker 的
        # 实际处理帧率（受 skip / suspect 模式影响）。初始化时可用标称值，
        # 运行时由 AIWorker 定期回传实际 effective_fps 来动态修正。
        self._effective_fps = config.AI_FPS  # 初始标称值，运行时动态更新
        self._confirm_time_s = config.GUARD_CONFIRM_TIME_S
        self._clear_time_s = config.GUARD_CLEAR_TIME_S

    @property
    def _confirm_frames(self) -> int:
        return int(self._confirm_time_s * self._effective_fps)

    @property
    def _clear_frames(self) -> int:
        return int(self._clear_time_s * self._effective_fps)

    def update_effective_fps(self, fps: float):
        """由 AIWorker 调用，更新实际处理帧率以保持帧数折算准确。"""
        self._effective_fps = max(1.0, fps)
        # 清空判定过滤门槛
        self._clear_min_conf = config.GUARD_CLEAR_MIN_CONF
        self._clear_min_area = config.GUARD_CLEAR_MIN_AREA

    async def process_frame(self, detections, frame):
        """每帧调用，驱动状态机。由 AIWorker 调用。"""
        self._last_frame_time = time.monotonic()

        # 检查人工接管
        # 注意：can_guard_send() 需要在 ControlArbiter 中新增实现，
        # 参照现有 can_auto_track_send() 的模式，检查当前控制权是否允许
        # GUARD_MISSION 发送命令。实现时一并补上。
        if self._control_arbiter and not self._control_arbiter.can_guard_send():
            if self._state not in (GuardMissionState.STANDBY, 
                                    GuardMissionState.MANUAL_OVERRIDE):
                self._abort_mission("人工接管")
                self._state = GuardMissionState.MANUAL_OVERRIDE
            return

        # 根据当前状态分发处理
        if self._state == GuardMissionState.STANDBY:
            self._on_standby(detections)
        elif self._state == GuardMissionState.GUARDING:
            self._on_guarding(detections)
        # DEPLOYING/RETURNING 由 MotionScriptRunner 异步驱动，
        # process_frame 中不做额外处理
        # RETURNING 期间即使检测到人也不重新出动，先完成回位

    def _on_standby(self, detections):
        """待命状态：检查系统就绪条件 + 用 ZoneService polygon 判断是否有入侵。"""
        # 系统就绪前置条件
        if not self._check_system_ready():
            self._intrusion_counter = 0
            return

        has_intruder = self._check_zone_intrusion(detections)
        if has_intruder:
            self._intrusion_counter += 1
        else:
            self._intrusion_counter = max(0, self._intrusion_counter - 2)
        
        if self._intrusion_counter >= self._confirm_frames:
            self._start_deploy()

    def _check_system_ready(self) -> bool:
        """检查系统就绪条件，全部满足才允许触发任务。"""
        # 1. 不在 E_STOP
        if self._control_arbiter and self._control_arbiter.is_e_stop_active():
            return False
        # 2. 没有人工接管
        if self._control_arbiter and self._control_arbiter.is_manual_override_active():
            return False
        # 3. 冷却期已过
        if time.monotonic() - self._last_mission_end_time < self._cooldown_s:
            return False
        return True

    def _on_guarding(self, detections):
        """驱离状态：用全画面 person 检测判断是否已清空（驱离点视角）。
        
        清空语义：驱离点当前视野方向内持续无满足门槛的 person，视为处置完成。
        属于工程近似，不等同于地面真实区域的严格几何清空。
        """
        has_person_in_view = self._check_any_person_qualified(detections)
        if not has_person_in_view:
            self._clear_counter += 1
        else:
            self._clear_counter = 0
        
        elapsed = time.monotonic() - self._guard_start_time
        if (self._clear_counter >= self._clear_frames 
                and elapsed >= self._min_guard_duration):
            self._start_return()

    def _abort_mission(self, reason: str):
        """中止当前任务。任务直接作废，不从中间状态恢复。"""
        if self._script_task and not self._script_task.done():
            self._cancel_event.set()
        asyncio.create_task(self._stop_guard_audio())
        asyncio.create_task(self._control_service.handle_command("stop"))
        self._reset_mission_context()
        logger.info(f"[GuardMission] 任务中止: {reason}")

    def _reset_mission_context(self):
        """完整清零任务上下文，防止残留状态导致回到 STANDBY 后误触发。"""
        self._intrusion_counter = 0
        self._clear_counter = 0
        self._guard_start_time = 0.0
        self._script_task = None
        self._cancel_event = asyncio.Event()  # 重建，旧的可能已 set
        self._last_mission_end_time = time.monotonic()  # 进入冷却期

    def _check_zone_intrusion(self, detections) -> bool:
        """【STANDBY 专用】检查是否有 person 在 ZoneService polygon 内。"""
        for det in detections:
            if det.class_name != "person":
                continue
            x1, y1, x2, y2 = det.bbox
            anchor = ((x1 + x2) // 2, y2)
            if self._zone_service.is_inside_zone(anchor):
                return True
        return False

    def _check_any_person_qualified(self, detections) -> bool:
        """【GUARDING 专用】检查全画面中是否还有满足门槛的 person。
        
        过滤条件：
        - 置信度 >= GUARD_CLEAR_MIN_CONF（过滤低置信度误检）
        - 检测框面积 >= GUARD_CLEAR_MIN_AREA（过滤远处小人影）
        """
        for det in detections:
            if det.class_name != "person":
                continue
            if det.confidence < self._clear_min_conf:
                continue
            x1, y1, x2, y2 = det.bbox
            area = (x2 - x1) * (y2 - y1)
            if area < self._clear_min_area:
                continue
            return True
        return False
```

**与 AIWorker 的对接**：
- 在 `AIWorker._process_detection()` 中添加分支：如果 `GuardMissionService` 启用，将 detections 喂给它
- `GuardMissionService` 和 `AutoTrackService` 互斥，不同时启用

**与 ControlArbiter 的对接**：
- 在 `ControlOwner` 枚举中新增 `GUARD_MISSION`
- 优先级：`E_STOP > REMOTE_CONTROLLER > WEB_MANUAL > GUARD_MISSION > AUTO_TRACK`
- `GuardMissionService` 在 DEPLOYING/GUARDING/RETURNING 状态时持有控制权

**验收**：
- 用 SimulatedAdapter 测试全流程：STANDBY -> DEPLOYING -> GUARDING -> RETURNING -> STANDBY
- 验证状态切换日志正确
- 验证人工接管后任务直接终止，释放后回到 STANDBY（不恢复中间状态）
- 验证 RETURNING 期间不重新触发

### 阶段 4：驱离动作实现（0.5 天）

**目标**：在 GUARDING 状态通过外接音频播放器播放警告音进行驱离。

**驱离动作方案**：

机器狗到达驱离点后保持站立不动，通过外接 USB 音频播放器循环播放预录的警告音频文件，直到区域清空或驱离超时。

```python
# 进入 GUARDING 时：
async def _start_guard_audio(self):
    """启动警告音频循环播放。"""
    # 使用 aplay 或 ffplay 播放预录 WAV/MP3 文件
    # 循环播放直到 cancel_event 被设置
    self._audio_process = await asyncio.create_subprocess_exec(
        "aplay", "-D", "plughw:1,0", "--loop", str(self._alert_audio_path),
        stdout=asyncio.subprocess.DEVNULL,
        stderr=asyncio.subprocess.DEVNULL,
    )

# 离开 GUARDING 时（清空确认 / 超时 / 中止）：
async def _stop_guard_audio(self):
    """停止警告音频播放。"""
    if self._audio_process:
        self._audio_process.terminate()
```

**配置项**：
- `GUARD_ALERT_AUDIO_PATH`: 警告音频文件路径，默认 `assets/alert.wav`
- 音频设备由 Jetson 上的 USB 音频播放器提供，设备名通过 `aplay -l` 确认

**验收**：到达驱离点后开始播放警告音，区域清空或任务中止后停止播放。

### 阶段 6：异常处理与安全保障（0.5 天）

**目标**：确保异常场景不会导致机器狗失控。

**异常场景覆盖**：

| 场景 | 处理 |
|------|------|
| 运动脚本执行中视频流断开 | 立即 stop，进入 FAULT |
| 视觉链路停滞（FFmpeg 活着但无新帧超过阈值时间） | 视为视觉链路失效，立即 stop，进入 FAULT |
| 运动命令连续被拒绝（E_STOP / 限流） | MotionScriptRunner 返回失败，进入 FAULT |
| DEPLOYING 中人工遥控器介入 | ControlArbiter 接管，当前任务立即终止，进入 MANUAL_OVERRIDE |
| E_STOP 触发 | 任何状态立即 stop，进入 FAULT |
| 驱离超时（>120s 仍未清空） | 强制返回 |
| 后端进程崩溃重启 | 默认 STANDBY，不自动恢复中间状态 |

**视觉链路健康检测**：

不只看 FFmpeg 进程是否存活，还要检查最近 N 秒是否真的有新帧送达。如果 `process_frame()` 超过阈值时间（如 5s）未被调用，则视为视觉链路失效。

```python
# 在 GuardMissionService 内部定期检查：
async def _check_visual_health(self):
    if self._state in (GuardMissionState.DEPLOYING, GuardMissionState.GUARDING,
                        GuardMissionState.RETURNING):
        if time.monotonic() - self._last_frame_time > self._visual_timeout:
            self._abort_mission("视觉链路失效")
            self._state = GuardMissionState.FAULT
```

**安全规则**：
- 任务运行中 ControlArbiter 持有 GUARD_MISSION 控制权
- WEB_MANUAL / REMOTE_CONTROLLER 可随时抢占（人工优先）
- **人工接管一旦发生，当前驱离任务立即终止（脚本取消 + 音频停止 + stop），不尝试从中间状态恢复；操作员释放控制后系统回到 STANDBY**
- 所有运动脚本经过 ControlService 的 Watchdog 保护
- 脚本执行器感知命令应答，连续被拒绝时自动进入 FAULT
- FAULT 状态需人工在前端点击"重置"才能恢复 STANDBY

**验收**：模拟各异常场景，验证状态切换和 stop 指令下发正确。

## 12. 实施建议

基于当前代码，最合理的实施路线是：

**保留现有 YOLO 检测、ZoneService 像素区域判断、ControlService / ControlArbiter 命令链路和 B2 适配器；新增独立的 GuardMissionService 作为顶层任务编排器，在起点用 polygon 判断入侵、在驱离点用全画面检测判断清空，用参数化开环运动脚本实现"前往-驱离-返回"的闭环，状态机集中管理所有转换逻辑。**

这条路线的优点：

- **逻辑自洽**：正确处理了机载摄像头移动后视角变化的问题，不假装 polygon 在所有位置都有效；
- **边界诚实**：明确标注了清空判定是工程近似而非精确几何判断，并将"驱离点专用 polygon"设为第一优先升级项；
- **鲁棒性好**：系统就绪前置检查、命令应答感知、视觉链路健康检测、RETURNING 不重新触发、人工接管即刻终止；
- **可追溯**：每次任务 4 个关键时刻抓拍，形成完整证据链；
- **实现复杂度低**：没有标定、没有地面坐标、没有视觉标记，全部用现有能力组合；
- **工程风险小**：新增模块不改已验证的代码（AutoTrackService / ZoneService / ControlService）；
- **单人可执行**：预计 4~5 个工作日可完成 MVP 并上机验证；
- **可渐进升级**：MVP 跑通后，可按需加入驱离点专用 polygon 或闭环定位能力。
