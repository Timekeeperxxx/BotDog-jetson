# BotDog 视觉锚点防区驱离需求文档与实现思路

## 1. 文档目的

本文档用于定义 BotDog 机器狗在现有系统基础上新增的**视觉锚点防区驱离能力**，并给出与当前工程架构兼容的实现思路、模块改造方案、状态机设计、关键算法边界与阶段落地计划。

本文档面向：
- 项目开发者
- 联调与上机测试人员
- 方案评审与汇报使用者

---

## 2. 项目背景

当前 BotDog 系统已经具备以下基础能力：

- 基于 RTSP 视频流的 YOLO 人体检测
- 基于像素 polygon 的重点区域判断
- 基于 `ControlService` 的统一命令下发
- 基于 `ControlArbiter` 的控制权仲裁
- 基于 `UnitreeB2Adapter` 的宇树 B2 底层动作控制
- 基于 `AutoTrackService` 的视觉跟踪闭环
- 基于 WebSocket 的事件广播和状态可视化

现有系统适合“目标检测 + 目标跟踪”场景，但不直接适用于“固定防区驱离 + 精准返航”场景。纯开环脚本式前进/后退虽然实现简单，但返航误差会随往返次数累计，不适合要求较高的重复复位任务。

因此，需要在当前系统基础上新增一种更稳的方案：**视觉锚点防区驱离方案（Visual Anchor Deterrence）**。

---

## 3. 核心目标

在不引入雷达、SLAM 和地面坐标建模的前提下，实现如下闭环：

1. 机器狗在起始观察位置待命；
2. 机载摄像头持续监测前方防区；
3. 当检测到人员进入防区并持续确认后，触发出击；
4. 系统锁定防区对应的背景锚点，前进过程中不盯着人追，而是盯着该视觉锚点推进；
5. 当人员与防区锚点的重叠关系持续下降到阈值以下时，判定驱离成功；
6. 返航阶段不再靠“后退固定秒数”，而是基于视觉锚点的位置与尺度回到初始观察位；
7. 回到起始观察位后停止并恢复待命姿态。

---

## 4. 方案定位

### 4.1 这不是什么
本方案**不是**：
- 自主导航
- 雷达定位
- SLAM 建图
- 地面坐标求解
- 全局路径规划
- 纯时间开环往返

### 4.2 这是什么
本方案是：
- **基于视觉锚点的视觉伺服**
- **基于背景参照的驱离推进**
- **基于外观回归的视觉返航**
- **固定场景、固定观察方向、固定任务边界下的半闭环驱离方案**

---

## 5. 设计原则

### 5.1 明确排除项

| 排除项 | 说明 |
|--------|------|
| 激光雷达 / SLAM | 当前阶段复杂度过高，不纳入 MVP |
| 地面坐标建模 | 当前方案不求解真实世界坐标，只依赖图像锚点 |
| 视觉标记（ArUco/AprilTag） | 当前路线优先尝试“环境背景作为锚点”，不先引入人工标记 |
| 直接修改 `AutoTrackService` 为总控 | 其职责是跟踪目标，不适合承载独立驱离业务 |
| 单纯靠固定时间返航 | 误差累积不可接受，不作为最终返航方案 |

### 5.2 必须坚持的原则

1. **控制目标从“人”切换为“防区锚点”**  
   前进与返航的参考对象不是人，而是触发时记录下来的防区背景锚点。

2. **驱离成功靠“人与锚点重叠关系变化”判定**  
   而不是继续盯着人追到底。

3. **返航靠视觉回归，不靠时间镜像**  
   即：返回原观察位的依据是锚点的位置、大小和相对形状回到初始状态。

4. **与现有控制链保持一致**  
   所有动作仍然走：
   `GuardMissionService -> ControlArbiter -> ControlService -> RobotAdapter`

5. **异常时宁停勿冲**  
   一旦锚点丢失、视觉链路失效、人工接管或 E-STOP，立即中止任务并停机。

## 6. 关键术语定义

### 6.1 防区（Defense Zone）
由操作者在视频画面上配置的一块重点区域，多边形形式表示，用于触发驱离任务。

### 6.2 视觉锚点（Visual Anchor）
在告警触发瞬间，从防区所在位置提取并锁定的一组背景外观特征，作为后续前进和返航的视觉参照。

### 6.3 重叠关系（Overlap Relation）
YOLO 检测到的人体区域与锚点区域之间的重叠面积或重叠比例，用于判断目标是否仍处于防区影响范围内。

### 6.4 视觉返航（Visual Homing）
通过比较当前锚点与初始锚点在画面中的位置、尺度和形状差异，控制机器狗回到原来的观察位。

## 7. 系统工作流

### 7.1 待命阶段（STANDBY）

#### 行为
- 机器狗处于待命位，建议使用蹲坐或稳定站立姿态；
- 机载摄像头持续采集画面；
- YOLO 检测 `person`；
- 使用当前已配置的防区 polygon 判定人的脚点是否进入防区。

#### 触发条件
满足以下条件时，确认入侵成立：
- 检测类别为 `person`
- 人脚点与防区 polygon 重叠
- 连续持续达到入侵确认时间阈值

#### 关键动作
在入侵确认的瞬间：
- 记录防区的初始图像位置和尺寸信息
- 初始化视觉锚点跟踪器
- 保存锚点的初始状态（基准帧）

### 7.2 出击阶段（ADVANCING）

#### 行为
- 机器狗起立；
- 启动驱离模式；
- 不再盯着人追，而是使用视觉锚点作为控制目标；
- 根据锚点在画面中的位置进行小幅转向与前进控制；
- 同时通过喇叭播放警告音。

#### 控制思想
使用视觉伺服进行推进：

- 锚点偏左 -> 左转微调
- 锚点偏右 -> 右转微调
- 锚点太小 / 还远 -> 前进
- 锚点位置、尺度达到前进阶段期望 -> 停或缓行

#### 驱离成功判定
在每一帧中同时获得：
- YOLO 的 `person` 检测框
- 视觉锚点跟踪器输出的锚点区域

若满足以下条件并持续若干秒，则判定驱离成功：
- 人与锚点的重叠比例持续低于阈值
- 或画面中无满足门槛的 `person`
- 且至少经过最短驱离持续时间

#### 保护条件
为了安全，必须设置：
- 最大前进时间
- 最大允许转向角速度
- 近距离安全停止阈值
- 锚点丢失即停机

### 7.3 返航阶段（RETURNING）

#### 行为
- 不再依据“动态前进了 X 秒，就后退 X 秒”
- 改为持续使用视觉锚点做闭环返航
- 通过后退 + 小幅转向微调，让锚点逐步回到初始基准状态

#### 返航判定依据
返航过程中持续计算当前锚点与初始锚点之间的差异，至少包含：

1. **位置误差**  
   当前锚点中心与初始中心的差异

2. **尺度误差**  
   当前锚点面积与初始面积的差异

3. **形状误差**  
   当前锚点宽高比、边缘形态或相对布局与初始基准的差异

当这三类误差连续若干帧满足阈值时：
- 发送 `stop`
- 再发送 `sit` 或恢复待命姿态
- 返回 `STANDBY`

## 8. 状态机设计

建议新增 `GuardMissionState`，不与 `AutoTrackState` 混用。

```text
STANDBY
  └─(人进入防区并持续确认)→ LOCK_ANCHOR
LOCK_ANCHOR
  └─(锚点初始化成功)→ ADVANCING
ADVANCING
  ├─(驱离成功)→ RETURNING
  ├─(锚点丢失)→ LOST_ANCHOR
  ├─(人工接管)→ MANUAL_OVERRIDE
  └─(E-STOP / 视觉失效 / 命令失败)→ FAULT
RETURNING
  ├─(视觉归位成功)→ STANDBY
  ├─(锚点丢失)→ LOST_ANCHOR
  ├─(人工接管)→ MANUAL_OVERRIDE
  └─(E-STOP / 视觉失效 / 命令失败)→ FAULT
LOST_ANCHOR
  └─(人工处理 / 重置)→ STANDBY
MANUAL_OVERRIDE
  └─(释放人工控制)→ STANDBY
FAULT
  └─(人工重置)→ STANDBY
```

## 9. 模块改造方案

### 9.1 新增模块

#### `backend/visual_anchor_tracker.py`
职责：
- 锚点初始化
- 锚点更新
- 锚点丢失判定
- 锚点位置 / 尺度 / 质量输出
- 为前进和返航提供伺服误差

#### `backend/guard_mission_service.py`
职责：
- 驱离任务状态机
- 与 `AIWorker` 对接
- 与 `ControlArbiter` 对接
- 根据锚点误差生成推进与返航控制决策
- 控制警告音播放与停止
- 管理任务证据链

#### `backend/guard_mission_types.py`
职责：
- 定义任务状态
- 定义任务事件
- 定义锚点状态 DTO
- 定义判定结果 DTO

#### `backend/visual_servo_controller.py`
职责：
- 将锚点误差映射为 `forward / backward / left / right / stop`
- 支持 PID 或简化 P 控制
- 约束最大速度和最小命令间隔

### 9.2 修改模块

#### `backend/workers_ai.py`
修改点：
- 在现有 YOLO 检测链后，新增对 `GuardMissionService.process_frame()` 的分发
- 使 GuardMission 与 AutoTrack 保持运行态互斥

#### `backend/main.py`
修改点：
- 装配 `GuardMissionService`
- 装配 `visual_anchor_tracker`
- 注册相关 API
- 注册相关事件广播

#### `backend/control_arbiter.py`
修改点：
- 新增 `ControlOwner.GUARD_MISSION`
- 设定优先级：
  `E_STOP > REMOTE_CONTROLLER > WEB_MANUAL > GUARD_MISSION > AUTO_TRACK`

#### `backend/config.py`
新增配置：
- 锚点质量阈值
- 锚点丢失时间阈值
- 驱离最短持续时间
- 最大前进时间
- 返航位置阈值、尺度阈值、形状阈值
- 警告音频路径

## 10. 与现有模块的关系

### 10.1 不修改 `AutoTrackService` 的原因
`AutoTrackService` 当前职责明确，是为“目标跟踪”服务，而本方案的核心是：
- 防区触发
- 锚点推进
- 锚点返航
- 定点驱离

两者在业务语义上不同，因此应保持独立，避免状态机和控制逻辑互相污染。

### 10.2 继续复用的模块
以下模块保持复用：
- `AIWorker`
- `ZoneService`
- `ControlService`
- `ControlArbiter`
- `UnitreeB2Adapter`
- `EventBroadcaster`
- `AlertService`

## 11. 核心实现思路

### 11.1 防区触发
1. YOLO 检测到 `person`
2. 取检测框底部中心作为脚点
3. 使用 `ZoneService` 判断脚点是否在防区 polygon 内
4. 持续达到确认时间后触发驱离

### 11.2 锚点初始化
1. 在触发瞬间截取防区附近背景区域
2. 提取一组稳定背景特征
3. 保存初始：
   - 锚点中心位置
   - 锚点面积
   - 锚点布局特征
4. 启动锚点跟踪器

### 11.3 前进控制
1. 每帧更新锚点状态
2. 计算锚点相对期望位置的偏差
3. 将偏差送入伺服控制器
4. 生成离散运动命令：
   - `left`
   - `right`
   - `forward`
   - `stop`

### 11.4 驱离成功判定
1. 计算 `person` 与锚点区域的重叠关系
2. 重叠比例持续低于阈值，或连续无有效 `person`
3. 达到持续时间要求后，切换到 `RETURNING`

### 11.5 返航控制
1. 每帧更新锚点状态
2. 计算当前锚点相对初始基准的误差：
   - 中心误差
   - 面积误差
   - 形状误差
3. 将误差送入返航伺服控制器
4. 生成：
   - `backward`
   - `left`
   - `right`
   - `stop`
5. 当误差连续满足阈值时完成归位

## 12. 风险点与边界条件

### 12.1 关键风险：锚点遮挡
如果目标人物长时间完全挡住防区背景，跟踪器可能：
- 丢失锚点
- 漂移到人身上
- 导致返航偏差

#### 对策
- 不使用单一大框作为锚点
- 优先设计为多子锚点或多特征锚点
- 引入锚点质量评分
- 锚点质量连续下降时立即进入 `LOST_ANCHOR`

### 12.2 算力负担
当前系统已经运行 YOLO 检测，再引入锚点跟踪与视觉伺服，会增加算力负担。

#### 对策
- 优先使用轻量级背景锚点跟踪
- 避免在全分辨率下执行过重计算
- 将锚点处理限制在 ROI 附近
- 上机测试时先观察 AIWorker 帧率变化

### 12.3 不适用场景
本方案不适用于：
- 背景纹理过弱、锚点不可区分的场景
- 锚点长期被完全遮挡的场景
- 路径中存在复杂障碍物的场景
- 需要厘米级高精度定位的场景

## 13. 任务证据链要求

一次完整驱离任务至少记录以下 4 个关键时刻：

| 时刻 | 记录内容 |
|------|----------|
| 入侵确认 | 初始人框、防区、锚点初始化画面 |
| 开始前进 | 驱离开始时刻画面 |
| 判定驱离成功 | 人与锚点重叠关系脱离的证据画面 |
| 返航完成 | 锚点回到初始位置后的归位画面 |

这些证据可复用现有抓拍与存储基础设施，用于：
- 调试
- 复盘
- 方案演示
- 误判分析

## 17. 验收标准

### 功能验收
- 人进入防区后可触发驱离
- 机器狗能够以锚点为参照推进
- 驱离成功后可切换返航
- 返航后能够回到接近初始观察位
- 回位误差明显优于纯时间开环返航

### 安全验收
- 锚点丢失立即停机
- 人工接管立即中止任务
- E-STOP 立即生效
- 视频链路失效立即中止任务
- 命令连续失败进入异常态

### 系统验收
- 前端能看到状态变化
- 关键事件能通过 WebSocket 广播
- 证据链抓拍完整
- 与 AutoTrack 保持互斥

## 18. 最终建议

如果你的目标只是尽快做一个演示版，纯开环脚本方案实现更快；  
但如果你的核心问题已经明确是“返航误差太大、无法稳定回位”，那么视觉锚点防区驱离方案更值得做。

这个方案的优势在于：
- 不依赖雷达或 SLAM
- 能用现有视频链路继续演进
- 能显著优于纯时间回退
- 与当前工程结构兼容

它的代价在于：
- 复杂度高于开环方案
- 必须认真处理锚点遮挡与漂移
- 需要增加视觉伺服与锚点质量判定模块

**综合判断：如果当前最大痛点已经从“能不能跑起来”变成“返航精度不可接受”，则建议切换到本方案作为下一阶段主路线。**
