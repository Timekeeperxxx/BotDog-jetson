# 验收测试清单 (Acceptance Test Checklist)

## 1. 目的

为智能巡检系统 v1.x 提供一套可执行的验收测试条目，覆盖核心业务用例与关键非功能指标，供开发完成后逐项勾选。

## 2. 基础环境前置条件

* 后端 FastAPI 服务已部署并可访问。
* GStreamer / WebRTC 媒体管线已按 `08_media_pipeline_design.md` 配置完成。
* 前端控制终端可通过浏览器访问。
* 机器狗本体或 MAVLink 模拟器可稳定输出心跳与姿态数据。

## 3. 用例 UC-01：远程巡检控制

**目标**：验证操作员可安全、流畅地遥控机器狗行进。

**步骤**：

1. 启动后端与前端，确认 `/api/v1/system/health` 返回 `status=healthy`。
2. 操作员登录前端（如启用登录）。
3. 前端与 `/ws/telemetry` 建立连接，状态栏显示“在线”。
4. 点击“启动任务”按钮，记录本次 `task_id`。
5. 推动手柄/键盘方向键，观察机器狗或模拟器移动。

**预期结果**：

* HUD 中姿态与位置随动作变化，无明显跳变。
* 从操作到画面反馈的主观延迟不超过 150ms（可人工感知 + 简单计时）。

**通过标准**：在 3 分钟连续操作中，无明显“抽搐”或失控现象，且无超过 1 秒的控制卡顿。

## 4. 用例 UC-02：异常温度告警与抓拍

**目标**：验证高温异常可触发完整的证据链（图片 + 坐标 + 时间戳）。

**步骤**：

1. 保证 ONNX 推理服务与红外模块正常运行，`thermal_threshold` 保持默认 60°C。
2. 在测试环境中模拟一段 `T_MAX > 60°C` 的数据流。
3. 观察后端日志是否记录 “Thermal high” 类事件。
4. 在前端等待告警弹出与抓拍卡片刷新。
5. 在“历史存档”页面调用 `GET /api/v1/evidence`，按 `task_id` 检索刚才的记录。

**预期结果**：

* 告警从触发到前端可见的时间 < 1 秒。
* `anomaly_evidence` 表中新增一条记录，包含 `event_type`、`confidence`、`file_path`、`gps_lat/lon`。
* 前端可点击缩略图查看原图。

**通过标准**：完整证据链可复现至少 3 次，字段无明显缺失。

## 5. 用例 UC-03：链路失效与 Failsafe

**目标**：验证链路断开时系统能快速感知，并进入保护状态。

**步骤**：

1. 在巡检过程中，手动断开无线链路或关闭 MAVLink 模拟器输出。
2. 记录断开时刻。
3. 观察后端 `heartbeat_timeout` 判定日志与系统状态变化。
4. 观察前端 UI 是否及时出现“通讯中断”提示。

**预期结果**：

* 在 `heartbeat_timeout`（默认 3 秒）内，后端切换系统状态为 `DISCONNECTED` 或 Failsafe 状态。
* 前端顶部出现明显的“连接断开/失联保护中”提示，并禁止新的移动指令下发。

**通过标准**：从断开到 UI 明确标示失联的时间不超过 3 秒，且在失联期间无法下发新的移动指令。

## 6. 用例 UC-04：视频延迟与重连

**目标**：验证端到端视频延迟达标，且在弱网或编码管线重启时画面可恢复。

**步骤**：

1. 在视频画面中放置一个物理秒表或可视计时器。
2. 对比“现场物理秒表时间”与“屏幕显示时间”，估算端到端延迟。
3. 人为制造小段网络抖动（限速/短暂阻断）。
4. 观察视频是否会长时间停在最后一帧。
5. 如遇中断，检查 Watchdog 是否在 2 秒内重启 GStreamer 管线，并在前端显示“重连中”状态。

**预期结果**：

* 正常网络下，端到端延迟 ≤ 200ms。
* 短暂抖动后，画面可自行恢复，且不会卡死在假画面。

**通过标准**：多次试验中，均满足延迟与重连要求。

## 7. 用例 UC-05：配置修改与限流

**目标**：验证后台配置修改和控制限流机制有效。

**步骤**：

1. 以 Admin 角色调用 `POST /api/v1/config` 修改：
   * `thermal_threshold`（例如从 60 调为 55）。
   * `control_rate_limit_hz`（例如从 20 调为 10）。
2. 重启或热加载后端（视配置项是否支持热更新）。
3. 再次触发高温告警，验证阈值生效。
4. 使用脚本模拟高于 10Hz 的控制消息发送速率。

**预期结果**：

* 新阈值确实改变告警的触发条件。
* 当控制频率超过新配置的上限时，后端丢弃部分包并返回 `CONTROL_ACK` 限流警告。

**通过标准**：配置修改对行为产生预期影响，且限流时系统保持稳定。

## 8. 非功能性指标抽检

在上述用例执行过程中，同时关注：

* **CPU/GPU 负载**：长期运行下是否保持在可接受区间。
* **内存泄漏**：执行多轮任务后进程内存是否异常增长。
* **日志完整性**：关键路径操作均有日志记录，便于排障。
