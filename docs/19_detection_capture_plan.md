# BotDog 检测与抓拍能力落地计划

## Context
基于《检测与抓拍技术栈》和《识别与抓取架构》的要求，需要输出一份可执行的落地计划，确保 AI 旁路识别、取证抓拍、告警推送与历史追溯在不影响主控视频链路的前提下实现。目标是边缘端低功耗运行、检测稳定、取证完整、告警实时。

## 目标与范围
- 构建旁路视频取流与 AI 推理管线（FFmpeg Pipe + YOLO）。
- 实现多阶段判定（稳定性校验 + 冷却消抖 + 优先级）。
- 完成取证抓拍（高清截图 + 元数据关联）。
- 实时 WebSocket 告警推送（含缩略图与置信度）。
- 证据历史管理与自动清理策略。
- 系统韧性（温控降级、任务态激活）。

## 推荐实施方案
### 1) 旁路视频取流与采样控制
- 以 RTSP 为取流协议，使用 FFmpeg 子进程输出 raw BGR24。
- 采用“高帧率取流 + 逻辑跳帧”方式实现动态升频：
  - FFmpeg 固定较高帧率（例如 10 FPS）持续输出。
  - Python 在巡逻态按 `每 5 帧处理 1 帧` 实现等效 2 FPS 推理。
  - 判定疑似目标时切换为“全速处理”状态（10 FPS 推理）。
- 严格旁路：与主 WHEP 控制视频解耦，不复用主链路解码器。

**关键配置（示例）**
```
ffmpeg -hwaccel auto -i rtsp://localhost:8554/cam \
  -f image2pipe -vcodec rawvideo -pix_fmt bgr24 -r 10 -
```

### 2) AI 推理与多阶段判定
- 推理模型：YOLOv8n/YOLOv10n（nano）。
- 加速：NVIDIA 优先 TensorRT，否则 ONNX Runtime。
- 判定流程（建议用状态机实现消抖）：
  1. 进入判定：连续 N 帧（例如 3 帧）检测到同一目标，才触发抓拍。
  2. 退出判定：连续 M 帧未检测到目标，才重置告警状态。
  3. 冷却/消抖：配合 `alert_cooldown`，避免同一地点重复记录。
  4. 优先级分级（人 > 垃圾 > 自定义障碍）。

### 3) 自动取证抓拍与元数据关联
- 触发告警后从原始帧抓拍高清图片。
- 使用 Pillow 将 raw 帧编码为 JPG。
- 记录并持久化：GPS、时间戳、任务 ID、姿态信息。
- 数据关联建议：
  - GPS：从 `MAVLinkGateway` 读取 `SystemStatusDTO.gps_lat/gps_lon`。
  - 任务关联：从 `StateMachine` 获取当前 `task_id`。
  - 入库：通过 `AlertService.handle_ai_event`（需新增）使用 SQLAlchemy Async 写入 evidence 表。
- 本地文件系统保存原图，数据库存路径与结构化元数据。

### 4) 实时推送与前端告警卡片
- WebSocket 广播 `ALERT_RAISED` 事件。
- 消息体包含：缩略图、目标类型、置信度、时间、位置。
- 前端在 IndustrialConsole 渲染卡片，并支持声光提示。

### 5) 证据历史与清理策略
- 时间线、地点、任务名筛选。
- 地图标注异常点。
- 按存储阈值与保留周期进行循环清理。

### 6) 系统韧性与任务态激活
- 只在“任务执行”态启用 AI；待机自动休眠。
- 监控温度，自动降帧或暂停推理。

## 关键文件路径（规划）
> 需根据现有代码结构定位后再落地修改
- 后端 AI Worker 生命周期：FastAPI lifespan / asyncio 子进程管理
- 推理与判定逻辑模块：YOLO 推理 + 多阶段判定
- 证据存储：SQLAlchemy Async + 本地存储路径管理
- WebSocket 广播：EventBroadcaster / ws event handler
- 前端告警卡片：IndustrialConsole 相关组件

## 复用与约束
- 复用现有 FastAPI 生命周期管理、WebSocket 广播机制、数据库与文件存储约定。
- 严格旁路，不触碰主控视频解码链路。
- 默认低帧率，按需升频。

## 验证与测试
- 手动验证：
  - 任务态启动/停止 AI 进程。
  - 模拟目标出现，确认稳定性判定与冷却有效。
  - 抓拍图像、数据库记录、GPS/任务信息完整。
  - WebSocket 在 500ms 内推送，前端告警卡片正常。
- 负载验证：
  - 低帧率巡逻 CPU/GPU 占用可接受。
  - 升频后仍不影响主控视频流畅度。
- 容错验证：
  - 高温降频或暂停生效。
  - 存储满时清理策略正常。
