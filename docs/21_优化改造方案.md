# BotDog 项目优化改造方案

> **版本**：v2.0  
> **状态**：审阅稿  
> **适用范围**：仅限 `/home/jetson/Project/BOTDOG/BotDog` 仓库内  
> **硬性约束**：
> - 不修改仓库外任何文件
> - 不修改 `data/` 中现有 JSON 的格式
> - 不修改任何会影响 ROS2 话题读写的契约
> - 不推翻已确认正确的业务逻辑，只做架构、稳定性、可维护性、可发布性收敛

---

## 1. 目标

把当前项目从“功能完整、可运行”提升到“可交付、可维护、可回滚、可审计”的产品级状态。

本阶段不追求新增业务能力，优先处理这些问题：

- 入口编排过重
- 全局状态和隐式依赖偏多
- 配置项跨度过大
- 前端页面级控制器过大
- 测试和回归门禁还不够稳
- 运维与日志信息分散

---

## 2. 不变项

以下内容视为产品契约，不做破坏性修改：

1. `data/` 下已有 JSON 文件的结构和字段
2. ROS2 话题名称、读写方向、消息语义
3. 已确认正确的业务逻辑，例如：
   - 建图启动前清理导航后台进程
   - 紧急制动、控制优先级、导航状态广播等既有行为
4. 现有 API 的外部语义，除非是兼容性增强

---

## 3. 代码现状判断

当前代码不是单脚本工程，已经具备基础分层：

- 后端有 `routes / services / repositories / schemas`
- 控制、状态机、导航任务、建图、鉴权、遥测都已拆成独立模块
- 前端已使用 hooks 和组件拆分部分逻辑

但仍有明显的产品化阻力：

- `backend/main.py` 仍承担大量系统编排职责
- 多处使用全局 getter、单例和运行时隐式依赖
- 配置、日志、控制、导航、建图边界还不够清晰
- 单个前端页面承担了过多流程控制
- 回归测试有覆盖，但还有脆弱点

---

## 4. 改造原则

1. 先稳后整
- 先保证启动、鉴权、控制、导航、建图的基础稳定性
- 再做结构收敛
- 最后做体验和运维完善

2. 契约优先
- `data/` JSON 不动
- ROS2 契约不动
- 对外接口尽量只做兼容性增强

3. 让依赖显式化
- 能注入的依赖尽量注入
- 能配置的边界尽量配置化
- 能局部持有的状态不要升格为全局单例

4. 把副作用放在明确边界里
- 启动时初始化
- 运行时读写
- 关闭时清理
- 互斥流程要显式表达，不要靠隐式副作用散落

---

## 5. 改造范围

### 5.1 后端入口与生命周期

目标：
- 降低 `backend/main.py` 的编排复杂度
- 让系统启动过程可读、可测、可追踪

涉及文件：
- [`backend/main.py`](/home/jetson/Project/BOTDOG/BotDog/backend/main.py)
- [`backend/config.py`](/home/jetson/Project/BOTDOG/BotDog/backend/config.py)
- [`backend/services_config.py`](/home/jetson/Project/BOTDOG/BotDog/backend/services_config.py)

计划：
- 将数据库初始化、默认数据初始化、遗留任务清理、状态机启动、遥测启动、控制启动等步骤继续向独立模块收敛
- 把启动摘要、失败摘要、降级摘要统一成稳定输出
- 清晰区分“必须成功才能运行”和“允许降级继续运行”的组件

预期收益：
- 更容易定位启动失败
- 更容易做分阶段初始化
- 更容易做回归测试和故障恢复

### 5.2 配置体系收口

目标：
- 让配置项从“大杂烩”收敛为“分域配置”

涉及文件：
- [`backend/config.py`](/home/jetson/Project/BOTDOG/BotDog/backend/config.py)
- [`backend/services_config.py`](/home/jetson/Project/BOTDOG/BotDog/backend/services_config.py)

计划：
- 按域划分配置：
  - 基础网络
  - 鉴权安全
  - 控制
  - 导航
  - 建图
  - AI
  - 视觉/点云
  - 运维路径
- 区分：
  - 启动时配置
  - 可热更新配置
  - 仅开发环境配置

预期收益：
- 降低配置理解成本
- 减少“看得到但不知道谁在用”的参数
- 更适合产品交付和部署排查

### 5.3 控制与状态边界

目标：
- 保持现有控制逻辑不变，但让状态和控制职责更清晰

涉及文件：
- [`backend/control_service.py`](/home/jetson/Project/BOTDOG/BotDog/backend/control_service.py)
- [`backend/control_arbiter.py`](/home/jetson/Project/BOTDOG/BotDog/backend/control_arbiter.py)
- [`backend/state_machine.py`](/home/jetson/Project/BOTDOG/BotDog/backend/state_machine.py)
- [`backend/api/routes/control.py`](/home/jetson/Project/BOTDOG/BotDog/backend/api/routes/control.py)

计划：
- 保持状态机、控制服务、控制仲裁器各自职责边界
- 明确控制拒绝、E-STOP、watchdog、限流的责任归属
- 减少路由层直接参与控制内部决策

预期收益：
- 安全策略更容易审查
- 控制链路更容易测试
- 以后增加控制源时更容易扩展

### 5.4 导航、建图、任务链路

目标：
- 让导航、建图、任务执行的互斥关系和流程边界更清晰

涉及文件：

## 6. 改造顺序

### 第一阶段：稳态保护
- 冻结契约
- 梳理关键路径
- 确认锚点可回滚
- 建立回归门禁

### 第二阶段：结构收敛
- 收拢后端入口职责
- 整理配置边界
- 明确控制、导航、建图责任边界
- 拆分前端页面级控制器

### 第三阶段：产品收口
- 强化测试覆盖
- 整理运维信息
- 统一错误和日志输出
- 做最终验收清单，落到 [docs/23_交付验收清单.md](23_交付验收清单.md)

---

## 7. 明确不做的事

- 不重写业务逻辑
- 不改变 `data/` JSON 格式
- 不调整 ROS2 话题契约
- 不做仓库外部环境改动
- 不引入破坏性 API 变更
- 不为了“更干净”而牺牲现有可用性

---

## 8. 需要审阅的点

请重点审阅以下问题：

1. 这个改造顺序是否合理
2. 哪些模块应该优先收敛
3. 哪些配置应该留在环境变量，哪些可以进入可热更新层
4. 前端页面拆分粒度是否合适
5. 测试门禁是否足够覆盖产品级风险

---

## 9. 执行方式

如果这份规划通过审阅，后续实施会遵循：

- 每次只改一个明确范围
- 每次改动后先验证关键测试
- 任何时候都保持可回滚
- 任何涉及契约变化的改动先停下来确认
